Help Centre

Manage the customer knowledge base — FAQs, categories, video guides, the platform-wide contextual tooltips, and help analytics — from one workspace, with every change publishing instantly to the storefront and no code change.

ℹ️
What this module controls

The Help Centre workspace is a full content-management system (CMS) for everything a customer sees on the public /help.html page: the searchable FAQ knowledge base, the category tabs, the "Video guides" grid, and the "Still need help?" contact block. It also edits the small ⓘ contextual-help tooltips that appear next to fields, buttons and metrics across the whole platform. Finally, it reports how customers use help — most-viewed FAQs, most-searched terms, unanswered searches and product-tour completion.

✅
How it saves

Edits are mirrored to your browser first (so the screen stays instant) and then persisted to the DOODLY Postgres database through /api/help/admin. A green "● Live" banner confirms the save reached the database and shows the current FAQ, category and video counts. The storefront Help Centre reads the same data live via /api/help/public, so a published change is visible to customers immediately.

Two things persist to the browser only

The Tooltips and Settings tabs store their overrides in the admin's own browser (local storage), not Postgres. They still take effect immediately for everyone who loads the page from that build, but they are not shared across devices the way FAQs, categories and videos are. Edit them from the machine you administer with.

⚠️
Deleting a category takes its questions with it

The confirm prompt reads "Delete this category and its questions?" for a reason — the FAQs filed under it disappear from the public knowledge base. Move any questions you want to keep into another category (via Edit on the FAQs tab) before deleting. On the storefront, an empty category is simply not shown.

💡
Canonical categories get the right icon automatically

The storefront maps well-known category slugs (getting-started, products, delivery, subscription, wallet, payments, bottle-returns, b2b, account) to a matching dairy icon even though the database stores a generic "info" icon. Naming a new category to match one of these slugs makes it inherit the correct icon.

ℹ️
Where these tooltips appear

DOODLY resolves the right help bubble for almost any element automatically — a status badge, a table heading, a button label, a KPI card or a known term — without needing markup on the page. Editing the copy here changes the bubble everywhere that element appears. The system is described as multi-language ready, and overrides are stored separately from defaults so a reset never loses the originals.

✅
Best Practice

Keep tooltip copy short, factual and jargon-free — it is read at a glance while someone is mid-task. Use the Example line for concrete formats (a code like DOODLY12345, an amount like ₹100) rather than stuffing them into the description. Disable tooltips on a module only when they clutter a densely labelled screen, not as a way to hide missing copy.

⛔
Reset discards your edits

"Reset knowledge base to defaults" replaces your current FAQs, categories and videos with the factory content. Anything you authored is lost. Use it only to recover a broken knowledge base or start fresh — never as a quick way to clear one bad entry (use Delete or Unpublish for that).

💡
To change the phone or email

The contact panel is intentionally read-only here so support details stay consistent everywhere. Update them once in Brand / CMS settings and every surface — the Help Centre, the floating help launcher, order emails — picks up the new values automatically.

ℹ️
Where the numbers come from

Opening an FAQ, running a search (and whether it found anything) and starting or finishing the product tour are all logged automatically as customers use the site. The backend keeps per-article view counts, groups search terms, isolates zero-result searches, and tallies tour events, so the tables reflect real behaviour rather than guesses.

✅
Best Practice — run a monthly content loop

Once a month, open Unanswered searches and write an FAQ for each recurring term, filed in the closest category and published. Then check Most searched keywords and make sure the top terms lead to a clearly-worded answer. This closes the gap between what customers ask and what the knowledge base answers.

Watch the banner colour

A green "● Live" banner means your FAQ / category / video edits reached the database. Do FAQ edits reach customers immediately?

Yes. Once the green "● Live" banner confirms the save, the storefront Help Centre reads the same published data, so browsing customers see the change on their next load — no deploy or code change.

What is the difference between Unpublish and Delete?

Unpublish hides a question from customers but keeps it in the admin list so you can bring it back with one click. Delete permanently removes it from the database with no undo. Prefer Unpublish for anything you might reuse.

Where do the tooltips I edit here actually show up?

Everywhere on the platform. The contextual-help engine automatically matches badges, buttons, table headings, KPI cards and known terms to a tooltip key, so editing the copy on the Tooltips tab changes that bubble on every screen it appears — customer, admin and portal pages alike.

Why can't I change the support phone number on the Settings tab?

Contact details are shown read-only so they stay identical across the whole product. Edit them once in Brand / CMS settings and the Help Centre, the floating help launcher and emails all update together.

What powers the "Unanswered searches" list?

Every customer search is logged with how many results it returned. Searches that returned zero results are grouped and ranked here, giving you a ready-made backlog of FAQs to write.
