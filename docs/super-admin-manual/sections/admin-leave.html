Leave

Raise, review and approve staff leave in one queue — with per-employee yearly balances, a two-step (manager then HR) approval path, and automatic marking of the approved days on the attendance register.

ℹ️
What this screen is for

Leave Management is the single place where all staff leave lives. A request can start two ways: HR raises it on an employee's behalf from this screen (New request), or an employee raises it themselves from Account → My HR → Leave (Apply for leave). Either way the request lands in the same queue here. From here you approve or reject it. On final approval DOODLY does three things automatically: it records the approval, deducts the days from that employee's yearly balance for the leave type, and marks each leave day on the Admin → HR → Attendance register. Everything runs against the live database through /api/admin/leave — there is no demo data.

⚠️
Who can do what

Viewing the queue is open to Super Admin, Admin and Accountant (Accountant is view-only). Creating a request and approving / rejecting require the leave module's edit right, which by default only Super Admin and Admin hold. The server backs this up: GET needs leave:view and POST needs leave:edit, otherwise it answers 403 Forbidden. When an approver who is Admin, Super Admin or Accountant approves, the server treats it as the final HR approval; a plain manager (granted leave:edit by a custom role/override) only performs the first step, moving the request to Manager approved for HR to finish.

How the day count works

DOODLY counts calendar days, inclusive of both ends — a Monday-to-Wednesday request is 3 days. It does not currently skip weekends or holidays automatically, so for staff who work six days a week the count is straightforward; for others, check the range you enter. The count you see is what gets deducted from the balance and marked on attendance.

On Submit request the form calls POST /api/admin/leave with action:"create". The server assigns the next code (LV-0001, LV-0002, …), stores the request as Pending, and — if the employee has a reporting manager on file — sends that manager a notification, "Leave request to review". The request now appears at the top of the queue. Nothing is deducted from any balance yet; that happens only on final approval.

Figure — from request to approval; rejection at any open stage ends the flow with a notified employee

Two-step approval, explained. When the approver is Admin, Super Admin or Accountant, the server treats the approval as the final HR approval — one click finishes it. Only when a plain manager (someone granted leave:edit through a custom role or per-user override, who is not Admin/Super Admin/Accountant) approves does the request move to Manager approved and wait for HR to finish it. In a default DOODLY setup, where only Admin and Super Admin can act, an approval finalises immediately. The on-screen toast tells you which happened: "Leave approved ✓" for a final approval, or "Sent for HR approval" for a first step.

⚠️
Loss of Pay is different

A Loss of Pay leave keeps no balance (there is nothing to deduct) and each day is marked as Absent on attendance — which is what feeds unpaid-day handling in Payroll. Every other type is marked as Paid or Sick Leave and reduces its balance. Choose the type carefully: it is the single field that decides whether the day is paid.

⛔
Caution — an approved or rejected request can't be re-decided here

Once a request is Approved or Rejected, the server refuses another decision on it ("This request is already decided.") and the Approve / Reject buttons disappear from that row. Approval also writes the attendance days immediately. If you approved by mistake, you must correct the attendance entries on the Admin → HR → Attendance register and adjust the balance separately — there is no one-click "un-approve".

Figure — the five balance buckets returned by the balances view (Loss of Pay is excluded)

Balances are read with GET /api/admin/leave?view=balances&employeeId=… (optionally with a year). The server returns, for the year, one row per type — allotted, used and remaining (never below zero). If an employee has no balance row yet, DOODLY shows the default allotment for that type (Casual 12, Sick 12, Earned 15, Maternity 180, Emergency 6). The row is actually created in the database the first time a leave of that type is finally approved. The balances view covers the five paid types only — Loss of Pay is deliberately excluded because it has no balance.

ℹ️
Where staff see their own balance

Employees see the same numbers on their Account → My HR → Leave tab, alongside their own request history and an Apply for leave button (type, from/to dates with a live day count, and an optional reason). Those self-service requests submit to /api/account/hr with action:"apply-leave" and appear in this admin queue as Pending, exactly like ones you raise here.

✅
Best Practice

Work the queue daily from the Pending filter, and always check the balance chip before approving — if someone is out of Casual balance, either approve knowingly or switch the type to Loss of Pay so the day is correctly unpaid. Give every rejection a clear reason: the employee receives it in their notification, and it is the only explanation stored. Because a final approval writes attendance and deducts balance in one irreversible step, confirm the dates and type are right before you click Approve.

Frequently asked questions

Does approving leave automatically update attendance?

Yes. On final approval DOODLY marks every day of the range on the attendance register — as Sick Leave for Sick, Absent for Loss of Pay, and Paid Leave for all other types — with a note referencing the leave code. You do not mark those days by hand.

How are the number of days calculated?

It counts calendar days inclusive of both the start and end dates, with a minimum of one day. Weekends and holidays are not skipped automatically, so a request spanning a weekend counts those days too.

Can an employee raise their own leave?

Yes — from Account → My HR → Leave using Apply for leave. It lands in this same queue as a Pending request for you to approve or reject.

What happens to the balance when I reject or cancel a request?

Nothing is deducted. Balances only change on final approval. A rejected request never touches the balance or attendance; it just records the reason and notifies the employee.

Why can't I change an allotment on this screen?

This module uses fixed default allotments (Casual 12, Sick 12, Earned 15, Maternity 180, Emergency 6; Loss of Pay has none). A balance row can carry a different allotted value once created, but there is no allotment editor on the Leave screen itself.
