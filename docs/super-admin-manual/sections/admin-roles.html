Roles & Permissions

The single control room where the Super Admin decides who can see and do what across the whole platform — set the defaults for a whole job role, fine-tune one single action, or grant an exception to one individual person, and see every change logged.

ℹ️
What this module does, in plain words

Every employee in DOODLY belongs to a role (Admin, Accountant, Operations Manager, Delivery Executive, and so on). A role is simply a saved bundle of permissions. This screen lets you (a) manage the list of roles, (b) tune exactly which actions each role is allowed on each part of the system, and (c) give one specific person a little extra — or a little less — than their role normally gets. The golden formula the platform follows is: Effective permission = the role's default  ⊕  any explicit grant on the role  ⊕  that person's individual override. The most specific setting always wins, and the person's own override beats everything.

⛔
Caution — this is the master key to the building

This is one of the most powerful screens in DOODLY. A wrong setting here can lock a whole team out of the tools they need, or hand someone access to money, customer data or the audit trail that they should never see. Because of that, only the Super Admin can open it. Every other role that navigates here sees a locked message: "Roles & Permissions is restricted — only the Super Admin can manage roles and user-level permission overrides." Read the screen carefully, change one thing at a time, and use the Audit tab and Run tests button to confirm the result.

✅
The three layers of access — remember this order

1. Role level (broad). A "level" for each module: None → View → Manage → Full. This is the everyday tool, edited mostly on the sibling Permissions page. 2. Granular role grant (medium). On the Role Matrix tab you can switch one exact action (say "Delete" on Reports) on or off for a whole role, overriding the level. These show in amber. 3. User override (narrow). On the User Permissions tab you grant or revoke one action for one person only. A user override always wins over the role default.

Tip — how this page relates to its two neighbours

Three System screens work together. User Management (/admin/users.html) is where you create staff accounts and pick their role. Permissions (/admin/permissions.html) is a simpler grid for setting each role's broad level (None/View/Manage/Full) per module. The green banner on this page even reminds you: edit the broad module levels on the Permissions page.

ℹ️
The permission "vocabulary" — 16 action types

Under the hood, DOODLY recognises 16 distinct actions: View, Create, Edit, Delete, Export, Import, Approve, Assign, Reassign, Cancel, Restore, Print, Download, Manage Settings, Manage Users, Manage Roles. Not every action applies to every module — the grid only shows the ones that make sense (for example Assign and Reassign appear on Delivery Mgmt, Drivers and Routes; Cancel and Restore on Orders and Subscriptions; Manage Roles only on this screen and Permissions). Seven of them — View, Create, Edit, Delete, Export, Approve, Print — are the "core" columns; anything extra shows under the Special column.

⚠️
The four "critical reserved" areas even Admin cannot fully touch

By design the ordinary Admin role is deliberately blocked from four sensitive areas so that no one but the Super Admin can quietly widen their own powers: Permissions (no access), and Settings, Roles and Audit Logs (view-only). This is baked into the default matrix and mirrored on the server. If you need someone to help manage roles, that person effectively needs to be a Super Admin — think hard before granting that.

ℹ️
"Surface" decides which app a role even sees

Before any module permission is checked, DOODLY checks the role's surface. Only admin-surface roles ever get the admin navigation or reach /admin/* pages. A Customer or Delivery Executive is denied every admin URL outright — no matter what the permission matrix says — and their admin navigation is empty. This is a hard rule in both the client (canRoute, filterNav) and the server (canAccessPath).

⚠️
Important — custom roles are permission templates, not fully assignable job titles

In the live database a user's role is stored as a fixed list (an enum) of the eleven built-in roles. That means a brand-new custom role you invent here is a reusable permission template the platform reads, but it cannot be permanently written onto a user account on the backend the way a built-in role can. Treat custom roles as a way to design and preview a permission set; for a role people will actually be assigned to long-term, prefer duplicating and shaping one of the built-in roles. This limitation is documented in the server code.

Screen note shown under the table: "Default roles can be duplicated into editable custom roles. Deleting a custom role reassigns its users to Customer Support."

ℹ️
How "level" and "explicit toggle" fit together

A role's baseline is its level per module (None / View / Manage / Full). View = just View. Manage = View + Create + Edit + Export. Full = all of those + Delete, plus that module's specials. On top of that baseline, a toggle you flip here writes an explicit grant or deny for one action that the system checks first. So you can, for instance, keep a role at "Manage" on Reports but explicitly add just the single "Delete" action — without promoting the whole module to Full. The amber highlight is your reminder that a cell is no longer simply inheriting.

✅
The precedence rule — a user override always wins

When DOODLY works out whether a person may do something, it checks: is there a user Grant? → yes, allow. A user Revoke? → yes, deny. Otherwise fall back to the role default (which itself already factors in any amber role toggle). That is why a single revoke on one person is the cleanest way to take one capability away from one employee without touching their whole team. The counters at the top-right — effective perms and overrides — update live so you can see the effect of what you just did.

⚠️
Prefer roles over piles of personal overrides

Individual overrides are perfect for a genuine one-off, but if you find yourself giving five people the same three grants, that is a sign you should make (or duplicate) a role instead. Scattered per-person exceptions are hard to audit and easy to forget. Keep the "overrides" counter low for each person wherever you can.

The "Run tests" self-check

A safe, one-click diagnostic that proves the access-control engine is behaving.

Purpose Let a Super Admin confirm — without any risk to real data — that inheritance, overrides and precedence all still work.
Where The Run tests button sits at the top-right of the tab bar; results appear next to it as a green (all passed) or red (some failed) badge.
💡
It never touches your live settings

When you click Run tests, the engine takes a snapshot of the relevant saved data, runs twelve checks on a copy, then restores everything exactly as it was. You'll see something like 12/12 tests passed; hover the badge to read each check by name. Use it any time you doubt a permission is applying correctly.

The twelve checks, in plain words

#What it verifies
1With no override, a user simply inherits their role's permission.
2A Grant override adds a permission the role does not have.
3A Revoke override removes a permission the role does have.
4A user override takes precedence over the role default.
5Setting a cell back to Inherit reverts to the role default.
6The "overrides" counter matches the number of active overrides.
7A granular role toggle enables an action for the whole role.
8A granular role toggle can also disable an action.
9Duplicating a role copies its permissions faithfully.
10Permission changes are written to the audit log.
11The "effective perms" count updates when an override is added.
12Applying the same override twice is safe (idempotent).

Best Practice — a safe change routine

1) Make one change. 2) Open User Permissions and use 👁 Preview as user to see the platform through an affected person's eyes. 3) Check the Audit tab shows your change with the right old → new value. 4) If in doubt, click Run tests. 5) Return to yourself. Small, verified steps beat a sweeping edit you can't easily undo.

Frequently asked questions

Quick answers for new administrators.

What is the difference between the "Permissions" page and this "Roles & Permissions" page?

The Permissions page is a simple grid for setting each role's broad level per module (None / View / Manage / Full) and saves those levels to the database. This page is the advanced control: it manages the roles themselves, lets you toggle a single action for a role, and lets you grant or revoke one action for one person. Use Permissions for the big strokes and this page for the fine detail.

If I change a role's permission, does it affect people already in that role?

Yes — a role is a shared bundle. Every person in the role immediately gets the new set of permissions (minus any personal overrides they have). That is exactly why roles are efficient, and why you should change them thoughtfully.

Can I make a role that can do everything except manage roles?

The built-in Admin role is already close to that — full access to nearly everything, but held out of Permissions, Roles, Settings-edit and Audit-Logs-edit. Those four areas are deliberately reserved for the Super Admin, so a role that manages roles is effectively a Super Admin.

How do I temporarily see what a delivery executive or accountant sees?

Go to User Permissions, pick that person, and click 👁 Preview as user. You'll be dropped into the platform with their exact navigation, allowed pages and buttons. Use the banner to return to yourself when you're done.

Is every change here recorded?

Yes. Every role creation, deletion, assignment, toggle and override is written to the audit trail with who, when and the old → new value — visible on the Audit tab here and in full on the Audit Logs screen.
