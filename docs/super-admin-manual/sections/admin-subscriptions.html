Subscriptions

The back-office register of every recurring milk plan on DOODLY — a live, searchable, exportable list of active and lapsed subscriptions, backed by a full management engine that lets an Admin create a plan for a customer and pause, resume, skip, edit, cancel, refund or re-mandate it.

ℹ️
What this module is

A subscription is a customer's standing order for milk — a chosen bottle (variant), a quantity, a delivery slot and a plan length (for example a 30-day or 90-day plan). The engine delivers automatically each morning, honours a strict 8 PM cut-off for changes, and respects skip and pause rules. This screen is where the office watches and manages all of them. It is a separate module from Subscription Billing (which handles the AutoPay charges) and from the customer's own My Subscription self-service page.

⚠️
Strictly Admin & Super Admin

Every subscription endpoint is guarded by requireSubsAdmin. Only the Admin and Super Admin roles may view or manage subscriptions — never a customer, Delivery Executive, Accountant, Manager, or even Support. Any other role receives a hard 403 Forbidden from the server; the on-screen filtering is only for convenience, the real security boundary is in the backend guard.

How the page comes alive

On load the page paints the recipe (header, four demo KPI cards, a table shell). Then wireSubscriptionsBackend() calls GET /api/admin/subscriptions?pageSize=200, maps each record into a table row, re-mounts the table and shows the green live banner. A second call to GET /api/admin/subscriptions/stats overwrites the four KPI numbers. If the backend is unreachable you get an amber/red banner and the demo numbers stay — so the layout is never blank.

Tip — the stats endpoint returns more than four numbers

Behind the four visible cards, /stats also returns total, cancelled, expired (completed + lapsed active/paused past their end date), autopayOn (active plans on AutoPay), renewalsDue7d (active plans ending in the next 7 days), mrrPaise (monthly recurring revenue = Σ daily price × 30 across active plans), trialCashback (credited count and amount) and newThisMonth. These power reporting and can be surfaced without new backend work.

Server-side filters also exist

The GET /api/admin/subscriptions endpoint itself accepts rich query parameters — status (with PAUSED folding in VACATION and a special EXPIRED bucket), autopay (on/off), plan, product, zone, a from/to start-date range, a text query q (id, name, email, phone), plus sort, dir, page and pageSize — and returns filter facets (plans, products, zones) for building a filter UI. The current storefront page loads the first 200 unfiltered and lets the client DataTable do the filtering; the same endpoint powers server-side filtering when needed.

Backed by the options endpoint

The create form is fed by GET /api/admin/subscriptions/options, which returns three data shapes depending on the query: with no query it returns active plans (id, name, days, discount) and subscribable products with their subscription variants (label, ml, daily price); with ?q= it searches customers by name / email / phone (max 10); with ?userId= it returns that customer's saved addresses (default first). This drives the customer picker → address picker → plan → bottle flow.

✅
Best practice — respect the 8 PM cut-off

Any change that lands after 8 PM takes effect a day later than you might expect. When a customer wants milk "from tomorrow", place the order before 8 PM; after the cut-off the earliest first delivery is the day after tomorrow. The system enforces this automatically, so trust the computed start date rather than back-dating.

⛔
Caution — cancellation and refunds

Cancelling sets the end date to now, turns off auto-renew and cannot be undone from this screen — a cancelled plan cannot be edited or re-paused; you would create a new subscription instead. If you enter a refund amount it is credited to the customer's wallet immediately (capped at ₹10,000) and appears in their wallet ledger. Double-check the amount before you confirm — a wallet credit is real money out of the business.

✅
Best practice — always give a reason

Pause, skip and cancel all accept a reason, and edits record a diff. Because every action lands in the timeline and the audit log with your role and the time, always type a short reason ("customer travelling 10–20 Jul", "wrong bottle size", "duplicate plan"). Six months later that one line is the difference between a clear history and a mystery.

Is "Subscriptions" the same as "Subscription Billing"?

No. This module manages the plans — who subscribes, to what, and the pause/skip/cancel lifecycle. Admin → Commerce → Subscription Billing is a separate module that handles the money — the Razorpay AutoPay mandates and recurring charges. The AutoPay toggle here flips renewal on/off and suspends the mandate; the billing module is where charge attempts and failures are worked.

Why does the table say "Order" when these are subscriptions?

The screen reuses the shared Orders table layout to save code, so the header labels read "Order / Customer / Items / Amount / Payment / Status". Every row is a real subscription — the "Order" cell is the short subscription id and the "Payment" cell shows AutoPay or Manual, not a payment record.

What does the "New (wk)" card actually measure?

The label reads "New (wk)", but the live number is new this month — the count of subscriptions created since the 1st of the current month. Read it as a month-to-date "new subscribers" figure.

Can I refund a customer when I cancel?

Yes. The cancel action accepts an optional refund amount (up to ₹10,000). It is credited straight to the customer's DOODLY wallet (not their card), recorded in their wallet ledger and logged as a "REFUND" event on the subscription timeline. Leave it blank to cancel without any refund.
