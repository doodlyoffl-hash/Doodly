CMS

The Content Management System — one place to edit the words, images, banners, FAQs, legal pages, testimonials and product details that appear on the public DOODLY storefront, so marketing copy and product launches can change without a code deploy.

ℹ️
Where this lives & what it really is

Open it from Admin → Content → CMS (URL /admin/cms.html). The page has two halves. The top half is a hub of six tiles that route you to each content area. The bottom half is the Content Blocks manager — a live table of every editable content block stored in the DOODLY database, driven through the endpoints /api/admin/cms (list & create) and /api/admin/cms/<id> (edit & delete). When it loads you will see a green banner reading “● Live — N content block(s) · P published from the DOODLY database”. If the banner is red, the backend is offline or your role is not allowed — see Troubleshooting.

ℹ️
Two content systems, one screen

CMS covers two kinds of content that are edited from different places but both reach the live site: (1) Content blocks — free-form JSON records of type page, banner, testimonial, legal, hero, faq and so on, stored in the CmsBlock table, published blocks are overlaid onto storefront pages at load time; and (2) Product content — every attribute of a product (pricing, variants, plans, nutrition, quality, description, badges, images, availability, SEO), edited through the tabbed Product CMS editor that opens from the Products screen. Both are documented below.

⛔
Caution — Delete is permanent and Marketing cannot do it

Deleting a block hard-removes the record from the database; there is no soft-delete or undo. Only Admin and Super Admin may delete — a Marketing user who tries gets a 403 (“Only Marketing / Admin can manage CMS”). If a page suddenly loses content, check whether its block was deleted rather than merely unpublished.

✅
Best Practice — draft, preview, then publish

Create or edit a block with Published off, confirm the JSON is right, then flip it on. Because only published blocks reach the storefront, this lets you stage copy safely. Keep keys descriptive and prefixed by area (page.about, banner.hero, legal.privacy) so the storefront's prefix-based hydration finds them.

ℹ️
Note — how blocks reach a page

Storefront prose sections (About, Farmers, Bottle-Returns, etc.) carry data-cms="<key>" wrappers. After the page renders, the site fetches the published blocks for each key-prefix on that page via /api/cms/page?prefix=<prefix> and overlays their eyebrow, heading, text and html fields onto the DOM. If the backend is unreachable it silently keeps the built-in copy — so a page never goes blank. This is why block keys are prefixed by area (e.g. about.hero, about.story).

Troubleshooting

SymptomLikely causeFix
Red banner “⚠ Your role can't view CMS (403)”.You are signed in as a role without cms:view (only Super Admin, Admin, Marketing have it).Ask a Super Admin to grant CMS access, or sign in with an authorised account.
Red banner “⚠ Backend offline — start next-app”.The Next.js backend that serves /api/admin/cms is not running or unreachable.Start / restore the backend, then reload the page.
Save fails with “A block with that key already exists.”Another block already uses that Key (keys are unique).Choose a different Key, or edit the existing block instead of creating a new one.
Save fails with “Data must be valid JSON.”The Data box contains malformed JSON (missing quote/brace/comma).Fix the JSON — every key and string value must be double-quoted; no trailing commas.
Delete fails with “Only Marketing / Admin (403).”Delete needs Admin/Super Admin; a Marketing user cannot delete blocks.Ask an Admin to delete it, or just un-publish the block to hide it instead.
Edited a block but the storefront still shows old copy.The block is saved as a Draft (Published off), or the page/section key-prefix does not match.Turn Published on; confirm the block Key's prefix matches the page's data-cms wrapper.
A product edit didn't stick / everything reverted.Someone clicked “Reset all CMS edits”, which clears every override globally.Re-enter the product edits; reserve Reset for a genuine full rollback.

Frequently asked questions

Do I need to redeploy the site after a content change?

No. Published content blocks are read at page load, and product edits apply to the live catalogue immediately. That is the whole point of the CMS — copy, banners, prices and product status change with no deploy.

What is the difference between “un-publish” and “delete”?

Un-publishing (Published off) keeps the block in the database but hides it from customers — fully reversible. Deleting removes the record permanently and is Admin/Super-Admin-only. Prefer un-publish unless you are sure the block is gone for good.

Why can I create and edit blocks but not delete them?

Your role is probably Marketing. Marketing has “manage” rights on CMS (view, create, edit, export) but not delete. Delete is reserved for Admin and Super Admin.

Can I change a block's Key later?

No. The Key is locked once a block exists (the field is disabled in edit mode) because the storefront looks blocks up by Key. If you need a different key, create a new block and delete the old one.

Where do I actually launch a new product?

On the Products & status tile → Products screen → the product's Edit → Product CMS editor → Basic tab → set Status to Active and Visible on site on, then Save.

✅
Best Practice — every change is audited

Creating, editing and deleting content blocks each write an audit entry (cms.create / cms.update / cms.delete) tied to the acting role. If live copy changes unexpectedly, the Audit Logs screen tells you who changed which block and when — use it before assuming a bug.
