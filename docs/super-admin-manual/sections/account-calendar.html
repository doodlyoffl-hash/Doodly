Delivery Calendar

A month-at-a-glance grid that shows the signed-in customer every delivery, skip and pause on their own account, painted directly from their real delivery records.

ℹ️
Where this screen lives

The Delivery Calendar is a customer self-service page, not an admin screen. Each customer sees only their own deliveries. It is documented here so a Super Admin understands exactly what the customer is looking at when they call in about a "missing dot" or a "wrong month". Route /account/calendar.html, manifest key account/calendar, block type calendar.

Two render modes — this matters for support

The page ships a static demo grid (a fixed "June 2026" with sample dots and a paused day) that renders for anyone exploring the site who is not signed in with a real account. The moment a genuine customer is signed in, a live routine (wireCalendar) replaces that whole panel with the customer's real month built from their delivery rows. If a customer says "the calendar shows the wrong month / fake dots", first confirm they are actually logged in — a logged-out browser only ever sees the demo grid.

What every day cell means

The live grid maps each delivery's status onto one day cell. Days with no delivery record stay blank (plain white cells). These are the only states the code produces:

The "Next delivery" outline only shows when today's cell is not already outlined

Today's forest outline takes priority over the green next-delivery box on the same cell. So when the next delivery is today, the customer sees the "today" outline rather than the "next delivery" box — this is expected, not a bug.

💡
Tip — the calendar has no filters, search, sort, export or pagination

Unlike the admin data tables in this manual, the Delivery Calendar is a pure visual grid. There is nothing to search, no columns to sort, and no export button. Navigation is limited to the ‹ and › month arrows. If a customer needs a line-by-line list they can export, point them to My Account → Deliveries (the Deliveries table) or their Invoices, not the calendar.

Data behind the calendar

The live grid is assembled from two backend calls that fire once when any account page loads (the account hydrator), then cached on the page. The calendar itself sends no requests when you flip months.

ℹ️
180-row window

/api/deliveries returns at most the 180 most recent deliveries. For a daily subscriber that is roughly the last six months. Flip far enough back and older months will simply show blank cells — the data has scrolled off the window, it was not deleted.

Best practice — fix the data, not the calendar

The calendar never invents anything — every dot is a real Delivery row. If a customer's calendar looks wrong (a delivered day showing amber, a missing green dot, or no "next delivery" box), the correct fix is in the operational data: update the delivery's status in Delivery Management or reschedule the subscription. Once the underlying Delivery record is corrected, the customer's calendar reflects it on their next page load. Do not chase it as a display bug.

FAQ

Can the customer change a delivery from the calendar (skip, pause, reschedule)?

No. The calendar is read-only. The only interactive controls are the ‹ and › month arrows. Skips, pauses and reschedules are done from My Subscription, and the results then show up here as amber (skipped) or moved dots.

Why does today's cell look different from the "next delivery" cell?

They are two separate markers. "Today" is a dark outline that always tracks the current date. "Next delivery" is a green outline box on the soonest upcoming delivery. When both fall on the same day, the "today" marker wins.

Does flipping to another month reload data or cost a request?

No. All the customer's delivery rows are loaded once when the account page opens. The ‹ / › arrows only re-draw the grid from that cached data, so month navigation is instant and offline-safe.

Can a customer see another customer's deliveries here?

Never. The /api/deliveries query is locked to the signed-in user's own subscriptions and orders, so the calendar can only ever contain that customer's own deliveries.
