Routes

Plan, name and optimise the delivery routes that carry DOODLY A2 buffalo milk from the dairy to each doorstep, zone by zone.

A route is a named driving plan for one delivery zone — a route name, an optional short code, an assigned delivery executive, and an ordered list of stops. In DOODLY there is no separate "stops" table: a stop is simply one of tomorrow's deliveries that has been tagged with this route (its routeId) and given a position in the running order (its sequence). Because of that, the Routes screen sits on top of the same live delivery data you see everywhere else in Operations — nothing here is invented or duplicated. The page shows a live seven-card KPI strip, the full list of routes, an overview map, and two pop-up dialogs — Plan a route and Manage route — that drive the whole life cycle: create, edit, assign a driver, re-sequence the stops with one click, activate/deactivate, and soft-delete. Everything reads and writes the production database through the /api/admin/routes* endpoints, so what you see is always the true state.

ℹ️
Note — a route is a plan, its stops are real deliveries

Creating a route does not create any deliveries. A route starts empty and fills up when the delivery-generation and assignment steps tag deliveries onto it. That is why a brand-new route shows 0 stops and a blank distance until it is used.

The live status banner

Directly under the strip, a coloured banner confirms the data is live: green reads "● Live — N route(s) from the DOODLY database" with the API base URL. If the backend is unreachable it turns red — "Backend offline … couldn't load live routes" — and if your role lacks access it shows "Your role can't view routes (403)". The screen deliberately shows the banner instead of stale numbers.

The stops list

Under the fields is a read-only list of the route's stops in delivery order. Its header summarises progress — e.g. "Stops (8/12 done · 15 bottles)" — where the counts come from the route's performance figures (completed vs. total deliveries, and empty bottles collected). Each line shows the stop's sequence number, the customer's name and their address. A ⚠ marker appears on any stop whose address has no map pin, with the hint "No map pin on this address — add one so it shows on the route map." If the route has no deliveries yet, the list reads "No stops on this route yet."

The route map

When the route's stops are geocoded, a map is drawn under the list with the stops as numbered pins in delivery order, joined by a line. If none of the stops have coordinates yet, the map stays hidden — geocode the customer addresses (from the address / serviceable-areas tools) to make it appear.

⚠️
Warning — optimise needs geocoded stops

Optimisation orders stops by their map coordinates. A route with no stops is rejected ("This route has no stops to optimise."), and stops with no map pin cannot be positioned by distance. Geocode the addresses first (look for the ⚠ markers in the stops list) so the sequence and the distance figure are meaningful.

✅
Best Practice — deactivate for a pause, soft-delete only when a route is retired

To take a route out of use for a while — a zone on hold, a seasonal round — clear the Active toggle instead of deleting it. That keeps the route, its driver and its stops intact and is instantly reversible. Reserve Soft-delete for routes you have genuinely retired; even then the record is only date-stamped, so an admin can restore it later.

Note — bulk endpoint

The platform exposes POST /api/admin/routes/bulk for batch moves — activate, deactivate, assignDriver, delete and restore across up to 1,000 routes at once (delete/restore require routes:delete). Every bulk run is audited. Day-to-day work on this screen is done per-route through the Manage dialog; the bulk endpoint is for scripted or future batch tools.

⛔
Caution — this is production routing

Client-side role hiding is only for tidiness — the real security boundary is enforced on every API call. Re-sequencing stops with Optimize rewrites the actual delivery order that drivers follow, and assigning a driver changes who runs the round. Treat these as live operational changes, not tests.

What exactly is a "stop"?

A stop is one of tomorrow's deliveries that has been assigned to this route and given a position in the running order. Routes do not have their own stop records — they borrow the live Delivery rows, so the stops list always matches the real deliveries.

Does creating a route schedule any deliveries?

No. A route starts empty. Deliveries are attached to it by the delivery-generation and assignment steps in Operations, after which they show up as its stops.

What does "Optimize stops" actually change?

It re-orders the stops into an efficient sequence (nearest-neighbour by map coordinates), rewrites each delivery's position, and stamps the route with a total distance in km and an estimated duration in minutes (roughly 20 km/h plus about 3 minutes per stop). It does not add, remove or reassign any deliveries.

Can I get a deleted route back?

Yes — soft-delete only date-stamps the record; it is not erased. An admin can restore it (the bulk endpoint offers a restore action), which brings the route and its settings back.

Why is there no search box or filter chips like on other lists?

This screen deliberately turns the DataTable toolbar and pagination off and shows the whole route list in one table (up to 300). You can still sort by clicking a column header. The backend does support search and filters — they are just not wired to a toolbar here.
