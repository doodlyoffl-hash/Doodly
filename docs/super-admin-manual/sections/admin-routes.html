Routes

Plan and manage the delivery routes your executives run each morning — group stops by zone, assign a driver, auto-optimise the stop order, and watch the route on a live map.

ℹ️
How routes are stored (no duplicate data)

A route does not keep its own copy of the stops. Each "stop" is simply one of today's Deliveries tagged with this route's ID and a sequence number. When you optimise a route, DOODLY only re-orders those deliveries and stamps the route's total distance and estimated duration — it never creates a second list to keep in sync. This is why a route's stop count, map pins and performance figures always match the real delivery board.

Overview map

Below the table sits a map preview. On load, DOODLY looks through the first few routes that have stops and draws the first one whose stops carry map pins (latitude & longitude). If nothing is geocoded yet, it keeps a 🗺️ card that reads: "Open a route below to see its delivery map — stops appear once their addresses are geocoded."

💡
Tip — get pins on the map

Route maps and distance optimisation only work for stops whose customer address has been geocoded (a saved latitude/longitude). Use the address pin-drop on the customer's address to add coordinates, and the stop will appear on the map automatically.

✅
Best Practice

Keep one route per zone per driver and give it a clear code (for example RT-VJ-02). Re-run Optimize stops after new subscriptions are added to the zone so the stop order and the estimated distance stay accurate for the morning run.

⚠️
Warning — optimise needs pins

Optimize stops refuses to run if the route has no stops ("This route has no stops to optimise."). It can only measure distance between stops that carry map pins — stops without coordinates are ordered but contribute 0 km, so a route with un-geocoded addresses may report a shorter distance than reality. Geocode the addresses first for a trustworthy figure.

ℹ️
Soft-delete, not permanent delete

Soft-delete simply hides the route from the live list and marks it Deleted; the record and its history are kept and can be restored. The manual delete on this screen never hard-deletes data.

Bulk operations exist behind the scenes

The backend also supports bulk actions on routes — activate, deactivate, assign a driver, delete and restore across many routes at once (POST /api/admin/routes/bulk, delete/restore need delete permission). The current Routes screen manages routes one at a time; bulk actions are available to tooling built on the API.

Is a "stop" a separate record I have to maintain?

No. A stop is just one of the day's deliveries linked to the route with a sequence number. You manage deliveries on the delivery board; the route only holds their order, distance and duration.

Why does turning a route Inactive not remove it?

Inactive keeps the route on file (grey badge) so you can re-activate it later — useful for seasonal or weekend-only routes. Use Soft-delete to retire a route you no longer plan to run.

How does "Optimize stops" decide the order?

It uses a nearest-neighbour method over the stops' map coordinates and straight-line (haversine) distance, then estimates time at roughly 20 km/h plus about 3 minutes per stop. It is a fast, sensible ordering — not a turn-by-turn routing engine — so always sanity-check unusual routes.
