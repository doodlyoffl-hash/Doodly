Serviceable Areas

The map of where DOODLY delivers — the list of pincodes you cover, the delivery charge, slot and ETA for each, the zone it belongs to, plus the customer waitlist for areas you have not reached yet. Add a pincode here and the storefront starts accepting orders from it the same instant, with no code change and no redeploy.

ℹ️
What this screen controls

This page is the single source of truth for delivery coverage. Whenever a customer types their pincode on the website — on the home-page checker, at checkout, or while adding an address — the storefront asks this list whether it can deliver there. A brand-new employee can think of it as: "the switchboard that turns delivery on or off for each pincode, and the sign-up sheet for people who want us in their area." Turn a pincode on and its customers can order; turn it off (or remove it) and checkout politely refuses and offers the waitlist instead.

ℹ️
Where the data lives

Every serviceable pincode is one row in the ServiceablePincode database table, grouped into DeliveryZone records (the same zones the routing and address systems use). The screen reads the list with GET /api/admin/delivery/pincodes and the zones with GET /api/admin/delivery/zones. Adding, editing, toggling and removing rows call POST, PATCH and DELETE /api/admin/delivery/pincodes — and each write is recorded in the audit log. The public storefront checks the very same table through GET /api/geo/serviceable?pincode=, which is why your changes go live instantly.

⚠️
The waitlist is stored in the browser, not the database

Unlike the pincode list, the waitlist capture (name, mobile, e-mail and locality of people in areas you don't yet cover) is saved in the visitor's own browser storage and mirrored into the admin panel. There is no server table for it. That is why the only thing you can do with it here is read it and Export CSV — always export before clearing browser data, or the entries are gone.

Live vs local

On open, the page loads the pincodes and zones from the database and shows a green banner: "● Live — N serviceable pincode(s) across N zone(s) from the DOODLY database (<api-base>)." If the backend cannot be reached it shows a red banner — "⚠ Backend offline … couldn't load live pincodes." — or, if your role is not allowed, "⚠ Your role can't view serviceable areas (403)." This manual documents the live, database-backed behaviour.

The Serviceable-areas table

Each row is one pincode and is inline-editable — you type straight into the cells and click Save areas to push every change to the database at once. The table is not paginated; it shows the whole coverage list, ordered by City then Pincode as returned by the backend.

AspectBehaviour on this screen
ColumnsPincode · Area · City · State · Zone (drop-down) · Charge (₹) · Slot · ETA · On (enable checkbox) · Delete (× button).
SortingFixed — the backend returns rows ordered by City (A→Z) then Pincode (ascending). There are no clickable column sorts on this table.
Search & filtersNo search box on the admin table itself. (The underlying GET /api/admin/delivery/pincodes endpoint does accept search, status = active/inactive/deleted and zoneId filters for API callers.)
PaginationNone — the full list is shown so you always see every covered pincode.
Row actionsEdit any cell inline · toggle the On checkbox to enable/disable delivery · click × to remove the row (takes effect on the next Save).
ExportsThe table has no export. Only the Waitlist panel exports (CSV) — see Screen 3.

💡
Disable vs remove

To pause a pincode (say a zone is temporarily out of stock), just un-tick its On box and Save — the row stays in your list and you can switch it back on later. Use the × Delete only when you genuinely stop serving an area. Deletes are a soft-delete on the server (the row is hidden and disabled but kept for history), so nothing is truly lost.

⛔
Caution — "Reset" is not just an undo

On the database-wired page, Reset clears the local working copy so the table falls back to the built-in seed list, and then runs the very same sync as Save over that seed list. In practice this can overwrite your live coverage with the seed defaults and remove pincodes that are not in it (deletes are soft, but the storefront still closes for those areas at once). If you only want to discard edits you have not saved, reload the page instead — that pulls a fresh copy from the database without writing anything.

⚠️
Operations can add and edit, but not remove

The Operations role has "manage" rights on Serviceable Areas — it can view, add, edit and enable/disable pincodes. It does not have delete rights. A DELETE from that role is refused with a 403. Only Admin and Super Admin can remove a pincode.

Friendly error messages

The modal turns backend errors into plain English on the inline error line: a duplicate reads "<pincode> is already serviceable.", a permission problem reads "Your role can't add pincodes (403).", and a connection problem reads "Backend offline — can't add right now." How people get onto the waitlist

When a visitor checks a pincode on the storefront and it is not serviceable, the checker shows a short form — Full name, Mobile, Email, Pincode and locality — under the message "Join the waitlist and we'll tell you the moment we launch near you." Name and Mobile are required. On submit they see "You're on the waitlist — we'll be in touch the moment we launch." Those entries are what appear here, grouped by pincode and sorted so the areas with the most demand are easy to spot.

Best Practice

Before adding a new pincode, decide its zone, delivery charge and slot up front so customers see consistent promises across an area — two pincodes in the same locality should share the same charge and window. When you retire an area, prefer disabling (un-tick On) over deleting so the row and its history stay available if you return. After every Save, confirm the green "Serviceable areas synced to the database" toast, then open the storefront checker in another tab and type the pincode to verify the customer-facing result matches what you set.

Frequently asked questions

Q: Does disabling a pincode cancel the orders already placed from it?

A: No. The On switch only decides whether new checkouts are accepted from that pincode. Orders and subscriptions already placed are unaffected — disable it and existing deliveries carry on while new orders from the area are turned away.

Q: If I delete a pincode by mistake, is it gone forever?

A: Not physically. Delete is a soft-delete — the row is hidden and disabled but kept in the database (with a deletedAt stamp) for history, and the server supports restoring it. Re-adding the same pincode is also always possible.

Q: The charge and slot I set here — do they actually reach checkout?

A: Yes. The public serviceability check returns the pincode's area, city, state, delivery charge, slot and ETA, so the customer sees exactly the values you entered and is charged the delivery fee you set.

Q: What does a Zone add beyond grouping?

A: A zone ties a set of pincodes to a delivery executive and to routes, and it drives the "Delivery Zones" KPI. Leaving a pincode with no zone still makes it serviceable; assigning one keeps your coverage organised for routing and reporting.
