Referrals

The customer's "refer a friend" screen — it shows their personal DOODLY code and share link, tracks the friends they invited, and pays a one-time wallet reward when a friend subscribes to a qualifying plan.

Note — this screen is read-only for the customer

The customer does not fill in a form here. Everything on this page is fetched from the server: the code, the share link, the four totals, and the friends list. The only editable control is the share-link box, which is read-only text kept there so the customer can select and copy it.

⚠️
Warning — a Trial Pack does not earn the reward

The reward is only released when the referred friend pays for a subscription whose plan length is at least the policy minimum (30 days by default). Trial Packs, single pours, 7-day plans and any shorter or free plan do not qualify. If the friend starts on a Trial Pack, their status stays "First Purchase Pending" until they upgrade to an eligible plan.

ℹ️
Note — one reward per friend, ever

Each referred friend can trigger the reward only once. Renewals or additional purchases by the same friend never pay a second time — the server keeps a single ReferralReward row per referred customer as the idempotency guard.

ℹ️
Note — how the code is applied (registration side)

The share link carries the code as ?ref=CODE. On the sign-up page the code is checked in real time by GET /api/referral/validate, which returns whether it is valid plus the referrer's first name (only the first name, for privacy). At registration the server re-validates it — see the field below — and links the new account to the referrer via User.referredById.

⚠️
Warning — self-referrals are rejected

If the referral code belongs to the same person signing up (matching email or phone), registration fails with "You can't use your own referral code." The admin console also blocks duplicate-account and circular referrals and logs them as fraud alerts.

Note — one shared policy

These three values drive the customer page's headline ("Earn ₹100 … 30-day or longer plan"), the stat tiles, and the automatic credit. Changing them in the admin Settings tab (saved via POST /api/admin/referrals, action config) instantly changes what every customer sees.

✅
Best Practice

Coach customers to share via the WhatsApp or Share buttons rather than typing the code by hand — the pre-built link already carries ?ref=CODE, so the friend's code is filled in automatically at sign-up and can't be mistyped. When helping a customer over support, confirm a "Pending" friend simply hasn't bought a qualifying plan yet; the reward will credit itself the moment they do — no manual action needed.

💡
Tip

"Wallet earned" on this screen and the wallet's referral credits always agree, because both are driven by the same WalletTxn rows (kind = referral). If a customer disputes their earnings, cross-check the Wallet page — the numbers are the same ledger.

Is the referral reward paid every time my friend renews?

No. It is a one-time credit per referred friend. Once their first eligible subscription pays out the ₹100, further renewals or purchases never pay again.

Can a customer change or regenerate their code?

Not from this screen — the code is fixed and shown as-is. An admin can regenerate a customer's code from the referral console if needed, which invalidates the old link.

Where does the reward money go?

Into the referrer's DOODLY wallet as a CREDIT of kind "referral". It appears both here under "Wallet earned" and on the Wallet page, and can be spent at checkout like any other wallet balance.

Why does the friends list only show first names / partial contact?

Privacy. On the customer screen only the friend's name and join date are shown. Full contact details, referrer codes and lifecycle actions live in the admin referral console, which is role-restricted.
