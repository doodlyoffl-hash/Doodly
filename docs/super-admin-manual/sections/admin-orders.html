Orders

A single, searchable list of every order placed across the whole DOODLY platform — who ordered, what they ordered, how much they paid, whether the money cleared and where the order is in fulfilment.

ℹ️
Where this screen lives and what it is

This is the staff-facing Orders list at route /admin/orders.html (manifest key admin/orders). It is assembled from just two blocks: a page head (title "Orders", sub-line "Every order across the platform.", with no action buttons) and a single table bound to the adminOrders dataset. Unlike the customer's My Account → Orders page — which shows only that one customer's orders — this screen shows all customers' orders in one place, pulled live from the DOODLY database via GET /api/admin/orders.

⚠️
This screen is read-only by design

The Orders list is a monitoring and lookup screen. There is no header button, no per-row "Manage" button and no inline edit on this page — you cannot create, edit, cancel, refund or delete an order from here. Those lifecycle actions happen elsewhere: a customer cancels/repeats from their own My Account → Orders; refunds are handled under Admin → Commerce → Payments; fulfilment moves under Admin → Operations → Delivery Mgmt; recurring plans under Admin → Commerce → Subscriptions. Use this screen to find and read orders, then jump to the right screen to act.

ℹ️
Live vs. demo data — read the banner

When the page loads it calls the backend and shows a status banner just above the table. A green "● Live — N order(s) from the DOODLY database (…)" banner means you are looking at real orders. If the backend can't be reached you'll see "⚠ Backend offline — couldn't load live data."; if your role lacks permission you'll see "⚠ No permission to view orders (403)." Any other error is shown verbatim as "⚠ <message>". Before you trust a count on this screen, confirm the green Live banner is showing.

"Items" is the order type, not the product list

Despite the column header, the Items cell on this admin screen shows the order type (Subscription, One time, Extra, Sample), not the line-item products. The backend does return richer detail (customer email & phone, the discount, the delivery date, the invoice number) that this table does not render — see "What the backend returns but the table hides" below. To read the actual products and money breakdown of a specific order, open it from the customer's My Orders view or the invoice.

Tip — the two pill columns answer different questions

Read Payment to answer "did we get the money?" and Status to answer "where is the order?". A healthy fresh order is often Paid + Out for delivery; a problem order to chase is Failed + On hold. Filter on those two facets together to build a work queue.

Table specification — the unified DataTable engine

Like every DOODLY list, the Orders table is upgraded by the shared DataTable engine (DOODLY_TABLE). It works over the order rows already loaded in the browser, so search / filter / sort are instant and need no extra server round-trips.

How the list loads

When the Orders route mounts, the front end calls GET /api/admin/orders?pageSize=200, maps each order into a table row, replaces the adminOrders dataset and re-mounts the DataTable, then shows the Live banner with the row count and the API base URL.

⚠️
Only the 100 most recent orders load

The front end asks for pageSize=200, but the backend caps the result at the 100 most recent orders (ordered newest-first) and ignores the pageSize value. So the table shows at most 100 rows and the on-screen search/filter only ever look within that window. If an order isn't in the list, it is most likely older than the newest 100 — narrow the query at the source (see the next note) rather than assuming it doesn't exist.

ℹ️
The backend supports server-side filters this screen doesn't use

GET /api/admin/orders accepts status (PENDING / PAID / FAILED / REFUNDED), type (SUBSCRIPTION / ONE_TIME / EXTRA / SAMPLE) and q (matches customer name, email or phone). The Orders page does not send these — it fetches the recent 100 and filters in the browser. For power lookups (e.g. every FAILED order, or all orders for one phone number) an admin can hit the endpoint with those query parameters directly.

ℹ️
Permission vocabulary vs. what this screen exposes

The orders module defines a rich set of actions in the permission matrix — view, create, edit, delete, export, approve, print, cancel, restore, download. That vocabulary controls order-related actions across the platform (for example a customer cancelling their own order, or export gating here). This particular Orders screen only ever exercises view (to load the list) and export (the Export button). It never creates, edits, cancels or deletes an order, regardless of how much access your role has.

Best Practice

Build a reusable Saved view for the two problem states you care about — e.g. filter Payment = Failed for a payments follow-up queue, or Status = On hold for stuck orders — and star the screen as a favourite. Because the active view also persists across reloads, your triage filter is one click away every shift. When you need everything (not just the recent 100) or a date range, export or query the API with status/type/q parameters rather than scrolling.

Can I cancel, refund or delete an order from this screen?

No. Orders is view-only. There are no create/edit/cancel/delete controls here. Cancellations happen from the customer's own My Orders page (within the allowed window); refunds are handled under Admin → Commerce → Payments. There is no hard-delete of orders anywhere in the product.

Why does the Status column sometimes show a delivery stage and sometimes "Active"/"Processing"?

When an order has a linked delivery, the Status pill shows that live delivery stage (e.g. Packed, Out for delivery, Delivered). When it has no delivery yet, the pill falls back to a label derived from the payment status: Paid → Active, Pending → Processing, Failed → On hold, Refunded → Refunded.

Does this screen include B2B orders and subscriptions?

This is the orders feed (subscription-generated, one-time, extra and sample orders). Recurring plans have their own screen at Admin → Commerce → Subscriptions, and business orders live under Admin → Commerce → B2B Orders. All three reuse the same DataTable engine, so search/filter/export behave identically.

How current is the data — do I need to refresh?

The list is fetched once when the page opens (the newest 100 orders). The ↻ Refresh button re-applies your current search/filter over the data already in memory; to pull fresh orders from the database, reload the page.
