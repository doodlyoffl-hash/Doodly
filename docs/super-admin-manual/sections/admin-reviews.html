Customer Reviews

See every product rating (1–5★) customers submit, moderate them one by one — approve, reject, hide, feature or reply — and rely on a hard server-side gate that lets only approved, verified, 5★ reviews ever appear on the public website, with every action written to the audit trail.

ℹ️
One page, one moderation module

The whole module lives at one screen (/admin/reviews.html, block type reviewsAdmin, mount #reviewsAdminMount, rendered by assets/js/reviews.js → DOODLY_REVIEWS.mountAdmin). It loads two calls in parallel — GET /api/admin/reviews?view=analytics and GET /api/admin/reviews?view=list — then paints a KPI strip and a single "All reviews" table with filters, search, per-row moderation buttons and a CSV export. Each part is documented below as its own screen.

✅
The public site is protected by policy, not by trust

Nothing you do in this module can accidentally leak an unfit review onto the storefront. The public feed (GET /api/reviews/public) hard-filters on the server to status = APPROVED and rating = 5 and a linked delivered order (orderId set = verified purchase). Pending, rejected, hidden, unverified, or 1–4★ reviews can never be shown publicly, no matter what. This module is for quality monitoring of everything; the storefront only ever sees your best, verified five-star feedback.

ℹ️
Where it sits in the menu

Although this manual groups Reviews under Growth, in the live admin sidebar the link appears under the Content section: Admin → Content → Reviews. The paths below use that real location.

These cards are read-only and count everything

There are no buttons on the KPI strip. The counts include pending, rejected and hidden reviews too — this is a quality-monitoring view of the raw feedback, deliberately wider than what the public sees. The strip refreshes whenever the module (re)loads, which happens after every moderation action.

💡

The current screen renders the totals and the 1–5 distribution; the extra fields are available in the API for reporting.

💡
Fast triage recipe

Set the Status filter to Pending and Rating to 5★ to see exactly the queue that could go public with one click — the approved-verified-5★ candidates. Then switch Rating to 1★/2★ to work the follow-up queue for unhappy customers. The Export button captures whatever slice you're looking at, so you can share a filtered batch with the team.

ℹ️
The server exposes more filters than the screen

The list endpoint also accepts productSlug and a from/to date range. The current screen wires up status, rating and free-text search; the extra parameters are available for API-driven reporting.

⚠️
Approving a 1–4★ review does not put its text on the storefront — but it does move the star average

The public testimonial cards only ever show 5★ reviews. However, the numeric product rating and 1–5 distribution shown on a product page are computed over all approved, verified reviews of any rating. So if you approve a verified 3★ review, that product's displayed average will drop accordingly (the stored Product.ratingValue/ratingCount are recomputed automatically). Approve honestly — approval is what makes a rating "count" toward the public score, even when the words stay private.

ℹ️
Every action is stamped and audited

Each moderation writes a moderator stamp — the acting role and user id (moderatedBy) plus a timestamp (moderatedAt) — onto the review, and appends an audit record such as review.approve whose target reads "<id> (PENDING → APPROVED)". You can always see who changed a review, from which status to which, and when, in Admin → System → Audit Logs.

💡
Reject vs. Hide

Both remove a review from public display and un-feature it. Use Reject for feedback you've decided is not acceptable (spam, abuse, off-topic). Use Hide to temporarily pull down a genuine review (for example while you follow up with the customer) with the intent to reconsider. The distinction is captured in the audit and lets you filter the two apart later.

ℹ️
Nothing here is permanently deleted

There is no delete button. Rejecting or hiding only changes a review's status; the record — and the customer's original words — is kept. This is deliberate: feedback is never destroyed, so a decision can always be reviewed or reversed by approving the review again.

ℹ️
Editing an existing reply

Opening Reply on a review that already has a response pre-fills the box with the current text, so sending again overwrites it. There's no separate "delete reply" button; to remove a reply, you'd replace it with new text.

✅
A reply is customer-facing — write it like a public statement

Because the reply is shown to the customer (and travels with the review), keep it warm, specific and on-brand. For a complaint, acknowledge the issue and say what you'll do — don't argue in the reply. Pair a reply with the right status decision (Hide while you resolve, then Approve if it's a genuine 5★).

⚠️
An approved review can quietly go back to Pending

Because a customer editing their review resets it to Pending, a review you approved yesterday can reappear in the Pending queue today if they changed it. Re-moderate edited reviews; the audit shows the customer's review.edit event.

ℹ️
Where approved 5★ reviews surface publicly

The homepage "What DOODLY families say" grid and each product page's "Ratings & Reviews" section read GET /api/reviews/public. Names are anonymised (first name + last initial, e.g. "Vivek D."), featured reviews sort first, then newest. If there are none yet, the site shows an honest empty state — never demo data.

Frequently asked questions

Why do I see 1–4★ reviews here if only 5★ ever go public?

This module is for quality monitoring of all feedback, not just the public showcase. You need to read every rating to spot problems, follow up on unhappy customers and keep the product average honest. The public feed applies its own hard filter (approved + verified + 5★) on top.

Can I permanently delete a review?

No. There is no delete. You can Reject or Hide it (which removes it from public view and un-features it), but the record and the customer's words are always kept, so any decision can be reversed by approving again.

Does replying to a review make it public?

No. A reply is saved and shown to the customer, but it does not change the review's status. Publishing still requires the review to be Approved, verified and 5★.

Who gets notified about a bad review?

When a customer submits a 1★ or 2★ review, all active Admins, Super Admins and Support staff receive an in-app "needs follow-up" notification pointing back to this module. The review itself stays untouched until you moderate it.

Which roles can moderate reviews?

By default Super Admin, Admin and Support (Customer Support) — the roles that hold the support permission. Viewing needs support:view; approving, rejecting, hiding, featuring and replying need support:edit. You can adjust these in Admin → System → Roles & Permissions.

How are customer names shown on the website?

Anonymised. The public feed shows only the first name and the last initial (for example "Vivek D."), never the full name, email or phone. Full identity is visible only inside this internal module.

✅
Best practice — a daily review routine

Once a day, open Admin → Content → Reviews and: (1) filter to 1★/2★ and reply to or resolve each, hiding while you follow up; (2) filter to Pending + 5★ and approve genuine ones, featuring the standout stories; (3) reject spam or abuse. Because everything is audited and nothing is deleted, you can always show exactly how each piece of feedback was handled.
