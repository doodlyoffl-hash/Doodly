My Orders

Every order the signed-in customer has ever placed with DOODLY — searchable, filterable and drillable — plus a live panel to rate delivered orders and earn loyalty points.

ℹ️
Where this screen lives

My Orders is a customer self-service page, not an admin screen. Each customer sees only their own orders — the backend scopes every query to the signed-in userId, so one customer can never see another's orders here. It is documented in this manual so a Super Admin knows exactly what the customer is looking at during a support call. Route /account/orders.html, manifest key account/orders. The page is assembled from three blocks: a page head, a table (dataset orders), and a reviewsPanel. To see or manage everyone's orders, a Super Admin uses Admin → Commerce → Orders instead.

⚠️
Two data modes — this matters for support

When nobody is signed in with a real account, the page renders the sample "exploration" dataset (mock orders) so the site stays browsable. The moment a genuine customer is signed in, a startup routine (hydrateAccount) wipes the demo seed and refills the table from the customer's real orders via GET /api/orders?limit=100. If a customer reports "these aren't my orders", first confirm they are actually logged in — a logged-out browser only ever shows demo data.

The status pill is computed on the client in a fixed priority order — the first rule that matches wins. There are exactly four possible pills:

1CancelledRedthe order has a cancelledAt timestampThe order was cancelled (by the customer or by the business).

ℹ️
How the data is loaded

On page start, hydrateAccount fetches GET /api/orders?limit=100. The backend caps limit at 50, so up to the 50 most recent orders (newest first) are loaded. All search, filtering and paging then happen in the browser over that set. The backend also supports server-side paymentStatus, type, date from/to, free-text q, cancelled and sort parameters, but the customer page relies on the client DataTable for those controls.

✅
Best Practice

When a customer calls about a specific order, ask them to read out the Order number (DOO-XXXXXX). It is the fastest key — it matches their row here, the order id in the backend, and the invoice. Then have them click that row to open the detail modal, which carries the timeline and self-service actions.

ℹ️
The backend returns more than the modal shows

GET /api/orders/[id] also returns the delivery driver's name and phone, the delivery address, the payment method, wallet transactions tied to the order, and any bottle-ledger entries for the delivery. The current customer modal displays the subset above; the extra data is available for support tooling and richer views.

⚠️
Cancellation is time-boxed

An order can no longer be cancelled once it is Out for delivery, Arriving soon or Delivered (or the linked delivery has moved to On the way / Reached / Delivered). The server enforces this and returns "This order is already on its way or delivered and can't be cancelled. Please contact support." — the Cancel button is also hidden in those states. There is no self-service hard delete of an order; cancelling is the terminal customer action.

💡
Tip — refunds land in the Wallet, not the card

Cancelling a paid order refunds the full total to the customer's DOODLY Wallet (a wallet credit), not back to the original card. If a customer expects a card refund, that has to be handled by support/admin — set the right expectation on the call.

ℹ️
"Rate" in the modal vs. this panel

The Rate button in the order detail modal (Screen 2) just adds a quick "Delivery rated" note to the order timeline. This Rate your deliveries panel is the real, verified-review flow: it is server-verified, moderated, feeds the public product rating and earns loyalty points. When a customer asks "why didn't my rating give me points?", they most likely used the modal Rate button — point them here.

ℹ️
One review per product

A customer can review each product only once. Re-buying the same product does not open a new review slot — they edit the existing review instead (which sends it back through moderation). The panel stays completely hidden when there is nothing reviewable and no past reviews.

✅
Best Practice

When encouraging reviews on a support call, remind the customer that points are awarded once per delivered order and only after the review passes moderation. Editing an approved review resets it to PENDING, so it temporarily drops off the public product rating until re-approved — set that expectation before they edit.

Can a customer delete an order from this page?

No. There is no delete. The only terminal customer action is Cancel order, and only while the order is still cancellable. A cancelled paid order is refunded to the wallet and shown as Refunded/Cancelled.

Where does a refund go when an order is cancelled?

To the customer's DOODLY Wallet as a credit (a refund transaction), and the order's payment status becomes REFUNDED. It is not returned to the original card automatically.

What is the difference between "Repeat order" and starting a new subscription?

"Repeat order" duplicates that exact order as a one-time PENDING order for immediate re-purchase. The header's "New subscription" button instead sends the customer to /subscriptions.html to set up a recurring plan.
