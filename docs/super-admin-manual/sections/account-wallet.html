Wallet

The customer's own money hub inside DOODLY — live balance, Trial Pack cashback, referral and bottle-deposit refunds, a secure "Add Money" flow, and a searchable record of every rupee that has ever moved in or out.

ℹ️
Where this screen lives

This is a customer-facing page. A signed-in customer opens it from the account sidebar under Bottles & money → Wallet. A Super Admin never edits money from here — staff use the separate Admin → System → Wallet Management screen (same DOODLY_WALLET engine, admin view) to credit, debit, refund, freeze or reverse. This manual page documents what the customer sees so support staff can walk a caller through it and read the same numbers.

What the page loads

For a real signed-in customer the panel first paints from the cached balance already in the account summary, then calls GET /api/wallet and re-renders with the authoritative ledger. Demo/guest visitors see a local sample wallet; a genuine signed-in customer never sees demo data — an empty or failed fetch shows a real (possibly zero) balance, never a seeded name.

ℹ️
The Trial Pack cashback rule (why "Pending" shows)

A customer who completed the paid ₹200 Trial Pack and then subscribes to an eligible plan (by default the 30-Day or 90-Day plan) is credited a one-time ₹200 to their wallet. The backend enforces this once per customer — TrialCashback.userId is unique, so even two payments at the same moment can only credit once. The credit fires automatically after the eligible subscription is paid; a refunded Trial Pack disqualifies it. Amount, eligible plans and on/off are all admin-configurable (no code change).

⚠️
Two different limit numbers — read carefully

The storefront config defaults to min ₹100 and max ₹1,00,000 with presets ₹100–₹5,000. The backend recharge config (appSetting "wallet.recharge") defaults to minPaise 10000 (₹100), maxPaise 1,00,00,000 (₹1,00,000) and preset paise values. The server number is the one that actually gates a real payment — if the two ever disagree, the customer sees a clear "Minimum/Maximum recharge is ₹X" error from the server. Keep them aligned when you change limits.

⛔
Caution — the client can never mint balance

In production, credits happen only through the signature-verified /recharge/confirm path. The plain POST /api/wallet/recharge route refuses unverified credits when the gateway is configured (returns use_gateway). A repeated confirm with the same payment id returns the original credit instead of adding money twice. On localhost a simulated payment path exists for demos only.

ℹ️
Recharge blocked states

If an admin has switched recharge off, the dialog shows "Wallet recharge is currently unavailable." If the specific wallet is frozen, it shows "Your wallet is temporarily frozen. Please contact support…" and no payment can start. Freezing/unfreezing is done from Admin → Wallet Management.

💡
Tip — what the customer gets notified about

A fresh top-up, a Trial Pack cashback, a referral reward and a bottle-deposit refund each fire an in-app notification, and (where the customer has opted in) a WhatsApp message — the recharge and cashback also send a branded email. Idempotent replays never re-notify, so a caller who refreshes will not get duplicate messages.

✅
Best Practice

When a customer says "I paid but my balance didn't change", ask for the Razorpay payment id shown on the success screen (or their bank statement). Search that id in Admin → Wallet Management: because every credit uses the payment id as its unique reference, you can instantly tell whether the credit landed, and if the gateway charged but confirmation failed, the amount is auto-refunded by the gateway — never manually credit a second time without confirming the first did not post.

FAQ

Q. Can a customer spend more than their balance at checkout?

No. When the wallet is applied to an order the server takes the smaller of the wallet balance and the order total, and the debit is idempotent per order, so a retry can never over-spend.

Q. Why does the same cashback amount appear under both "Lifetime cashback" and "Trial Pack cashback"?

Both tiles read the cashback credit total. Trial Pack cashback is the only kind of cashback DOODLY issues today, so the two figures match by design.

Q. Does wallet money expire?

Not currently. The config has an "Expiry (days)" setting and the reports expose an "expired credits" field, but the expiry policy is future-ready and left blank (never expires) by default.

Q. Can support add money to a customer's wallet directly?

Yes, but not from this customer page. Staff use Admin → Wallet Management to post a Manual Credit, Manual Debit, or Bottle Deposit Refund — each is audited and appears in the customer's history with the matching source label.
