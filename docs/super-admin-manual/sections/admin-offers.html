Offers

A campaign-driven promotion engine — create typed, prioritised, scheduled discounts that either apply automatically at checkout (codeless) or by code, then watch them earn revenue live through ten KPI cards and a single searchable, filterable table.

ℹ️
What this screen is, and how it is built

The Offers item under Admin → Growth opens the storefront admin page /admin/offers.html. That page renders a single block, offersBoard, which mounts into #offersMount and is driven entirely by the live DOODLY database through the API /api/admin/offers. Everything you see — the KPI cards, the filters, the table, the New offer and Manage forms — is real data; when the backend is reachable a green “● Live” banner confirms it and names the database. All money is typed and shown in ₹, but the backend stores every amount as integer paise (₹1 = 100 paise), so a 10% offer is stored as discountBps = 1000 and a ₹50 offer as flatPaise = 5000.

ℹ️
Offers vs Coupons — same maths, more power

Offers deliberately reuse the Coupons discount engine for all the arithmetic and eligibility rules (percent/flat, caps, minimum order, first-order, product/category/plan targeting, usage limits). What Offers add on top is a type (Percentage, BOGO, Bundle, Festival, Cashback and more), a numeric priority, a campaign name for roll-up reporting, and a full Draft → Active ⇄ Paused → Archived lifecycle. The other key difference: an offer’s code is optional. Leave it blank and the offer is codeless — it is resolved and applied automatically at checkout with no code to type.

ℹ️
Who can view and who can manage

Two permission levels guard this module. Viewing (the board, KPIs, table and analytics) is open to any role that can view offers, reports or customers — that is Super Admin, Admin, Marketing, and read-only viewers such as Accountant, Operations, Procurement, Quality, Inventory and Support (through their reports/customers access). Managing (create, edit, activate, pause, resume, archive, delete, restore, duplicate, bulk) requires the offers edit permission, which by default only Super Admin, Admin and Marketing hold. A blocked mutation returns HTTP 403 and the screen shows “Only Marketing / Admin can manage offers (403)”. The server-side checks canViewOffers / canManageOffers are the real boundary — hidden buttons are convenience only.

ℹ️
Status is derived — you set a state, the system decides the badge

You control an offer’s state (Draft / Active / Paused / Archived), but the coloured status badge on the table is computed from that state together with the scheduling window and the deletion flag. The rule, in order: deleted → Deleted; state ARCHIVED → Archived; state DRAFT → Draft; state PAUSED → Paused; a start date still in the future → Scheduled; an end date already passed → Expired; otherwise → Active. So an offer set to Active with a start date next week correctly shows as Scheduled until that date, then flips to Active on its own, and to Expired after its end date — no manual toggling required.

Tip — more analytics are computed than are shown

The dashboard endpoint also returns a conversion rate (discount ÷ revenue as a %), offers used today, and the average discount per redemption. Per-offer detail (a full redemption log with customer, order, discount and order total, plus unique-customer count) and campaign/offer report roll-ups are available from the same API (?view=detail and ?view=reports, with date presets Today, Last 7, Last 30, This Month, Financial Year, All). Generating a report writes an audit line via the log action.

Best Practice — always name a campaign

Filling the Campaign field on related offers (e.g. “Monsoon 2026”) costs nothing but unlocks the Active Campaigns KPI and the by-campaign report roll-up, so you can measure a whole promotion, not just one offer, and compare campaigns side by side.

The API stores a little more than the form shows

The offer record and its validation engine also support a description, a banner text/image for storefront display, category targeting (applicableCategorySlugs), a per-customer usage limit (perCustomerLimit), and a customer-specific eligibility list (eligibleUserIds when eligibility is SPECIFIC). These fields exist on the backend and are honoured at checkout but are not exposed on this admin form, which focuses on the common promotion controls above.

Tip — activate, pause, resume and archive from one dropdown

This form manages the whole lifecycle through the single Status dropdown: pick Active to launch, Paused to switch off temporarily, Archived to retire, Draft to pull back into editing — then Save changes. (The backend also exposes dedicated activate / deactivate / pause / resume / archive actions and a bulk action over many offers at once for API and future multi-select use.)

⚠️
Warning — only Active, in-window offers reach customers

An offer set to Draft, Paused or Archived, one whose start date has not arrived, one past its end date, or one that has hit its usage limit will never apply at checkout — even if its code is typed. If a customer says a promotion “isn’t working”, check the Status badge and the schedule first.

✅
Best Practice — build in Draft, schedule the launch, cap the risk

Create the offer as a Draft, review the numbers, then set it Active with a future Start date so it goes live on its own. Always set a Max discount on percentage offers and a Usage limit on aggressive ones so a single promotion can never give away more than you planned. Give competing codeless offers distinct priorities so the intended one wins.

⛔
Caution — a live code cannot be reused elsewhere

Offer codes are globally unique. If you need the same wording on a new offer, either duplicate the existing one (the copy gets a “-COPY” code you can rename) or archive the old offer first. Deleting is a soft delete — the offer and its redemption history are kept and can be Restored — but the code stays reserved until then.

Frequently asked questions

Q: What’s the difference between an Offer and a Coupon?

They share the same discount engine and rules. Offers add a type, a priority, a campaign name and a lifecycle, and — crucially — can be codeless so they apply automatically at checkout. Use Coupons for a plain typed code; use Offers when you want auto-apply, campaign reporting, or several promotions competing by priority.

Q: If two offers both qualify for a cart, which one wins?

The one with the higher Priority. If priorities tie, the one giving the larger discount wins. Set distinct priorities to control the outcome.

Q: How do I run a limited festival promotion safely?

Create it as a Draft with type Festival and a campaign name, set the Discount type/Value, add a Max discount cap and a Usage limit, set the Start and End dates, then set Status to Active and save. It goes live on the start date and expires on its own.

Q: I deleted an offer by mistake — is the data gone?

No. Delete is a soft delete: the offer and its redemption history remain in the database. Open it (list deleted offers) and click Restore to bring it back as a Draft.

Q: Does editing an offer change orders that already used it?

No. Past redemptions are recorded with the discount that was given at the time; editing only affects future checkouts.
