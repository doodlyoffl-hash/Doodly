Drivers

Onboard, monitor and manage the delivery executives who carry DOODLY A2 buffalo milk from the dairy to the customer's doorstep.

The Drivers module is the single place where you keep the delivery fleet in order. Every driver here is a real login account (role Delivery Executive) paired with a Driver record that stores their employee ID, vehicle, assigned zone and daily bottle capacity. The screen shows a live seven-card performance strip, a searchable and filterable list of executives, and two pop-up dialogs — Add delivery executive and Manage driver — that drive the full onboarding and life-cycle workflow. Everything on this page reads and writes the production database through the /api/admin/drivers* endpoints, so what you see is always the true state of the fleet.

ℹ️
Note

A driver record is not a stand-alone row — it is bonded to a User account. Adding a driver creates both a login (with the Delivery Executive role) and the Driver record in one atomic step. Suspending a driver locks that same login.

ℹ️
Note — the temporary password shows only once

After Reset password, the new secret appears in a toast such as "Temp password: Doodly@1234 — driver resets on next login". It is not stored in readable form and cannot be shown again. Copy it before the toast disappears, then hand it to the driver.

✅
Best Practice — suspend, don't delete, for temporary issues

For a driver who is on unpaid leave, under review, or has quit but may return, use Suspended (locks the login) or clear the Active toggle. Reserve Soft-delete for people who have genuinely left. Because soft-delete only stamps a date, an admin can still restore the record later, but suspend keeps their history and zone assignment intact and is instantly reversible.

Note — bulk endpoint

The platform exposes POST /api/admin/drivers/bulk for fleet-wide moves — activate, deactivate, assign zone, delete and restore across up to 1,000 drivers at once (delete/restore require drivers:delete). Every bulk run is audited. Day-to-day work on this screen is done per-driver through the Manage dialog.

⛔
Caution

Client-side role hiding is only for tidiness — the real security boundary is enforced on every API call. A driver's login and the customer-facing app depend on these records, so treat suspend, delete and password reset as production actions, not tests.

Does adding a driver here also create their login?

Yes. Saving the Add form creates a User account with the Delivery Executive role and the Driver record together, in one step. There is no separate account to create.

What is the difference between Inactive, Suspended and Soft-deleted?

Inactive takes the driver off the working roster but keeps the login usable. Suspended locks the login so they cannot sign in, while keeping them on the fleet. Soft-deleted removes them from the list entirely but preserves the record so it can be restored later.

Where do I set the driver's zone and capacity?

In the Manage driver dialog, not on the Add form. Zone feeds routing; daily bottle capacity feeds the auto-assignment engine.

Can I recover a driver I removed?

Yes — soft-delete only marks the record; it is not erased. An admin can restore it (the bulk endpoint offers a restore action), which brings the driver and their history back.
