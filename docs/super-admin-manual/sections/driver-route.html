Route & Maps

The delivery executive's road-map for the day — the ordered list of stops to visit, a map of the route, and one-tap turn-by-turn navigation to each customer's exact doorstep.

ℹ️
How this feature is built — two layers

"Route & Maps" exists in two places that share the same purpose. First, the Today's Route page inside the driver portal (/driver/route.html) is a lightweight sheet that lists today's stops and shows a map area. Second — and this is what an executive actually drives with in the field — the live executive portal at /delivery/ mounts the real map engine (DOODLY_MAPS) and the full route workflow (DOODLY_DELIVERY): an interactive route map, distance and ETA, and real Google Maps navigation links. This section documents both, and is honest about which parts are a placeholder and which are live.

⚠️
Not the admin Routes screen

Do not confuse this with Routes under Admin → Operations, where a manager plans and optimises route templates. This section is the executive-facing view: it only ever shows the signed-in executive their own assigned stops for the day.

When a signed-in executive opens the portal, it first shows "Loading your route…", calls the backend, and then draws the map and the stop list from real data. The map is provided by DOODLY_MAPS.routeMap(), which threads a line from the executive's current position through every stop that is not yet delivered and drops a numbered pin at each one; delivered stops are shown dimmed.

ℹ️
Note — distance & ETA are estimates

The distance on the meta bar is the straight-line (haversine) distance from the current position through the remaining stops. The ETA is a working estimate — roughly city speed of ~18 km/h plus about 3 minutes per stop. Actual turn-by-turn time comes from Google Maps once the executive taps a navigation link.

How the map is powered

Real Google Maps when a key is set; a built-in fallback map when it is not.

Purpose Make sure the route always draws — a proper Google map when the business has a Maps key, and a clean built-in map otherwise, with no broken tiles either way.
Access Super Admin configures the key; every executive benefits automatically. Path Environment key NEXT_PUBLIC_GOOGLE_MAPS_API_KEY, surfaced through GET /api/config

The map engine (DOODLY_MAPS) is written to mirror the Google Maps interface, so the same code runs with or without Google. It decides which map to show like this:

ConditionWhat the executive sees
Maps key is set — /api/config returns a mapsKeyThe real Google map, with Google's own tiles, markers and route line; it fits the view to include every stop.
No key — mapsKey is emptyA built-in, dependency-free map drawn as clean streets/parks with the same numbered pins and route line. Everything still works.
Key is rejected — billing off, wrong referrer, invalid keyGoogle's failure hook fires and the engine quietly falls back to the built-in map instead of showing a broken Google canvas.
✅
Best Practice

For the sharpest field experience, set NEXT_PUBLIC_GOOGLE_MAPS_API_KEY in the production environment and enable billing on the Google project. The navigation links open Google Maps regardless of the key, so even without a key executives can still route to every doorstep — the key only upgrades the in-app map from the built-in view to Google's tiles.

⚠️
Warning — never see someone else's route

An executive only ever loads their own stops. The backend looks up the Driver record linked to the signed-in user and returns deliveries for that driver alone; any non-executive role is refused. If an executive reports seeing another person's stops, treat it as a data/linking error and check their Driver-to-user mapping — do not dismiss it.

Note — payment shown per stop

A stop is labelled COD ₹{amount} only when its order is still pending and the method is cash; everything else shows Paid. This tells the executive up front which doors to collect money at — reconciled later on the Cash Collection screen.

Does the executive see other drivers' stops?

No. The route is scoped to the signed-in executive's own Driver record, and any non-executive role is refused. Everyone sees only their own run.

Do I need a Google Maps key for navigation to work?

No. Navigation links open the Google Maps app/site directly with the destination and travel mode, so they work without a key. The key only upgrades the in-app map from the built-in view to Google's tiles.

Which travel modes are supported?

Walking, Bike (bicycling), Scooter (two-wheeler) and Car (driving). Tap Navigate on a stop and choose the one you're using so the ETA matches.

What happens when I press "Start route"?

Every still-assigned stop moves to "On the way", the app begins reporting your location every 30 seconds so customers and admins can track you, and it opens navigation to your next stop. Location reporting stops automatically once every stop is delivered.
