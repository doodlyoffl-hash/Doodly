Quality Testing

The lab screen for milk quality — record a fat / SNF / density / temperature test on each collected batch, auto-grade it A/B/C against configurable acceptance rules, and approve or reject the batch so that only passing milk stays in raw-milk inventory.

ℹ️
How this screen is built (from the route manifest)

The admin/quality route (URL /admin/quality.html) stacks three blocks: a heading ("Quality Testing — Batch test results and cold-chain logs.") with two header buttons — Record test (primary) and Quality rules (ghost); a KPI row; and a data table bound to the quality dataset with its toolbar and pager switched off. On open, layout.js → wireQualityBackend() replaces the placeholder KPIs and the sample rows with live data from the DOODLY database. A "quality record" is not a separate table — it is a Procurement (the collected batch) plus an optional QualityTest. Grading thresholds live on a single QualityConfig row.

ℹ️
Who can open and change things here (real RBAC matrix)

Every action checks a quality permission on the server. In the authoritative matrix (lib/rbac.ts): Super Admin and Admin have quality: full; the dedicated Quality staff role has quality: full; the Procurement role has quality: view (can see records but cannot record tests, approve, reject or edit rules). "full" grants view + create + edit + export + delete; "view" grants read only. So viewing the list, stats, a batch detail and the rules needs quality:view; recording a test needs quality:create; approving / rejecting a batch and saving rules need quality:edit. A role without the right level gets a clear 403 message in the dialog.

What the screen loads

On open the screen calls GET /api/admin/quality, which returns quality (the enriched batch records, including untested/pending ones), stats and rules (the current acceptance thresholds). It then calls GET /api/admin/quality/stats for the dashboard numbers. The table shows the most recent 300 batches, newest collection first. Rows are built from the records — Batch = batch number, Farm = farmer name, Fat and SNF = the tested values (or the collection values before a test), Temp = temperature in °C (or "—"), and Result = a coloured status badge.

ℹ️
The backend can filter — the screen just doesn't expose it

The list endpoint GET /api/admin/quality accepts search, status (passed / failed / pending), farmerId, center, from and to query parameters. The Quality Testing screen calls it without any of these (so you always see the full recent list), but reports and integrations can pass them. A bulk approve/reject endpoint (POST /api/admin/quality/bulk) also exists on the server; it is not wired to a button on this screen today.

Tip — let the rules do the grading

When you pick a batch or edit Fat / SNF, the dialog compares them to the current acceptance rules and flips the Result select for you, showing "Rule: fat ≥ X%, SNF ≥ Y% → meets / below standard". Only override this if you have a lab reason to.

⚠️
A failed test removes milk from inventory

Saving a Failed result sets Procurement.accepted = false, and if that batch had already been accepted into stock it decrements the raw-milk (MILK_RAW) inventory by the batch litres in the same transaction. A Passed result records the test but does not change acceptance on its own — use Approve on the batch (Screen 3) to add it to inventory. The action is written to the audit log as quality.pass or quality.fail.

What it shows (from GET /api/admin/quality/{id})

The dialog first paints the batch summary already in memory, then fetches the detail. Grade is computed from the values against the current rules: C if fat is below the minimum fat or SNF is below the minimum SNF; A if fat ≥ (min fat + 1.5) and SNF ≥ (min SNF + 0.5); otherwise B. Inventory impact reads "In raw-milk inventory" when the batch is accepted, or "Excluded from inventory" when rejected. The history lists up to the farmer's 10 most recent tested batches so you can spot a pattern before deciding.

ℹ️
Idempotent by design

Approving a batch that is already accepted (or rejecting one already rejected) makes no change and no inventory movement — the server only adjusts stock when the accepted flag actually flips. So a double-click can't double-count the litres.

ℹ️
Saving is audited field-by-field

Save sends PATCH /api/admin/quality/rules. The server records which thresholds changed and writes a quality.rules.update audit entry listing the changed field names (prev → new). Changes apply to the next test immediately; already-recorded tests keep their original pass/fail. The rules live on a single shared QualityConfig row — there is one standard for the whole operation, not one per farmer.

✅
Best Practice

Test every batch the same day it is collected: record Fat / SNF / density / temperature while the batch is still Pending, let the rules suggest the result, and only override with a written reject reason. Approve passing batches promptly so raw-milk inventory reflects real stock, and review a farmer's history in the Manage dialog before accepting a borderline (grade C) batch. Change the acceptance rules rarely and deliberately — every save is audited and instantly re-grades all new tests.

Frequently asked questions

Q. Is a "quality record" a separate database table?

No. Each record is a Procurement (the collected batch) plus at most one QualityTest. There is no duplicate quality table — grading rules live on the single QualityConfig row, and acceptance is stored as Procurement.accepted.

Q. Does passing a test automatically put the milk into inventory?

No. Recording a Pass only saves the test. Adding the milk to raw-milk (MILK_RAW) inventory happens when you Approve the batch in the Manage dialog. A Fail, however, immediately rejects the batch and removes its litres if it had already been accepted.

Q. Can I change how strict the standard is per farmer?

No. The acceptance rules are one shared setting (QualityConfig) for the whole operation. You can view a farmer's individual history in the Manage dialog, but the pass/fail thresholds are global.

Q. Where do the dashboard numbers come from?

The header KPIs and the six-tile strip are computed live by GET /api/admin/quality/stats from the real QualityTest and Procurement tables — today's tests, 30-day averages, pending count, accepted/rejected counts and the compliance rate.
