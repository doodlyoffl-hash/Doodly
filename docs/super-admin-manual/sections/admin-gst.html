GST Management

One central, Super-Admin-controlled tax engine — create tax rates, assign them to products, let orders and invoices calculate GST automatically, and keep every historical invoice accurate even after a rate changes.

ℹ️
What this module does, in plain words

Instead of writing tax percentages into the code, DOODLY treats every GST rate as a configuration you edit on screen. You make a rate once (say "Dairy 5%"), attach it to the products it applies to, and from then on every order, subscription renewal, B2B order and invoice looks up that rate and does the sum for you. The clever part: whenever money actually changes hands, the system freezes a copy of the rate used on that transaction. So if you raise "Dairy 5%" to 9% next year, last year's invoices still correctly read 5%. Nothing historical ever silently changes.

⚠️
Who can change what

Every admin who can reach this screen can read the rates, product assignments, reports and audit trail. But only the Super Admin can create, edit, enable/disable, set-default, assign, delete, or change settings. For everyone else the action buttons disappear and the page shows a Read-only badge — "Read-only · Super Admin manages GST". On the backend this is enforced by the gst:config permission, which only the Super Admin holds.

💡
Tip — the "Run tests" button

At the top-right of every tab is a Run tests button. Clicking it runs the engine's own built-in checks (validation rules, the 5%-of-₹1000 = ₹50 sum, product resolution, and — most importantly — the "historical invoice stays unchanged after a rate edit" guarantee) and shows a green N/N tests passed badge. Hover the badge to see each check by name. It is a safe, read-only self-diagnostic: it snapshots your data, runs on a copy, and restores everything afterwards. Use it if you ever doubt the tax maths.

⛔
Caution — you cannot delete a rate that is in use

Deletion is deliberately protected. If a rate is assigned to one or more products, or appears on any recorded transaction, the delete is refused with a message like "Cannot delete — this rate is linked to transactions. Deactivate it instead." This safeguards your historical invoices. When a rate is retired, use Disable instead of Delete — it stops the rate being used going forward while preserving every past record. Only a rate that was never assigned and never used can be deleted, and even then a confirmation prompt ("This can't be undone.") appears first.

⚠️
Validation you will meet

If the form can't be saved, a red line appears above the Save button with the exact reason:

  • "GST name is required." — the name box is empty.
  • "GST percentage must be a number." — a non-number was typed.
  • "GST cannot be negative." / "GST cannot exceed 100%." — the percentage is out of the 0–100 range.
  • "An active GST rate with this name already exists." — you'd create a duplicate active name.
  • "'Effective until' must be after 'effective from'." — the end date is earlier than the start.

How a product's rate is decided (the resolution rule)

When any part of DOODLY needs a product's tax, it resolves the rate in this order:

  1. If the product is marked Tax-exempt → 0%, labelled "Tax-exempt".
  2. Else if it is assigned to a rate that is currently Active → that rate is used.
  3. Otherwise (unassigned, or assigned to a rate that has been disabled) → the Default rate is used.

This is why disabling a rate is safe: any product still pointing at it quietly falls back to the default rather than breaking.

Why the report can show zero on a fresh production site

Reports are built from real transaction snapshots only. Sample GST transactions are never seeded on the live site — they exist only for local exploration. So a brand-new production environment correctly shows empty reports until real orders and invoices start recording GST. This is expected, not a bug. In the DB-backed backend the same figures are aggregated from the live PaymentRecord, SubscriptionBilling and Invoice ledgers (each carrying a gstPaise snapshot).

Best Practice — always fill in "Reason for change"

When you edit a rate, take the extra second to type a reason (e.g. "GST Council notification 12/2026"). It is optional, but it is stored verbatim in the audit trail and turns a bare number change into a self-explaining compliance record your accountant and any future auditor will thank you for.

Settings save as you change them

There is no separate "Save" button on this tab — each control saves the moment you change it (and shows a "GST settings saved" confirmation). Changing the Default GST rate here has the same effect as clicking "Set default" on the Rates tab. The three roadmap toggles are placeholders for tax models that switch on when multi-jurisdiction billing is enabled; they are safe to leave off for a single-state operation like DOODLY today.

✅
Best Practice — trust the snapshot, edit rates freely

The single most important thing to understand about this module: editing a rate never rewrites history. When a sale happens, the exact GST percentage, amount and the rate's ID are frozen onto that transaction. So the correct way to handle a rate change (e.g. the GST Council moves ghee from 12% to 5%) is simply to Edit the existing rate to the new percentage — past invoices keep their old figure, and every future sale uses the new one automatically. You never need to create a duplicate rate or worry about "closing off" the old one. The built-in test "Historical txn keeps original GST % after rate change" exists precisely to guarantee this.

Frequently asked questions

If I change a rate, will old invoices change too?

No. Every transaction freezes the GST percentage and amount at the moment of sale, so past invoices are unaffected. Only future sales use the new rate.

How do I stop charging tax on a product?

On the Product GST tab, set that product's rate to Tax-exempt (0%). Its effective rate becomes "Exempt" and no GST is added on future orders.

What happens to a product I never assigned a rate to?

It uses the Default rate (shown at the top of the Product GST tab and set on the Rates or Settings tab). For DOODLY the seeded default is "Milk GST (Nil)" at 0%.

Why can Admins and Accountants see this page but not change anything?

Viewing helps finance reconcile and answer customer queries, but tax configuration carries compliance risk, so the ability to change rates is deliberately restricted to the Super Admin via the gst:config permission.

What's the difference between "GST" and "CGST + SGST" tax types?

Both charge the same total percentage. "CGST + SGST" simply splits that amount into two equal halves on the invoice (Central and State GST) — used for intra-state supply. The split display is also governed by the "Show CGST/SGST split" setting.

Where do the four starting rates come from?

DOODLY seeds four sensible dairy rates on first use: Milk GST (Nil) 0%, Dairy 5% (curd, paneer, palkova), Ghee & Butter 12%, and Standard 18%. You can edit, disable or add to these at any time.
