Products

The catalogue control room — the one place where you create products, price them, stock their variants, write their storefront copy, and flip a single status to launch or pull a product live, with no code change and no redeploy.

ℹ️
Where this screen lives and what it is

This is the staff-facing Products screen at route /admin/products.html (manifest key admin/products). It is built from three blocks: a page head (title "Products", sub-line "Manage the catalogue. Flip status to launch a product.", with two header buttons — Recommendations and Add product), a green notice banner about zero-code launch, and the product table (block type productAdmin). When the page opens it pulls the live catalogue from the DOODLY database with GET /api/admin/products?pageSize=200&sort=created and re-draws the table.

ℹ️
DOODLY sells one product today — but the catalogue is fully multi-product

The live storefront currently sells A2 Buffalo Milk; Curd, Paneer, Palkova and Ghee are staged as "Coming soon". This screen is not milk-only — it is a complete Product Information Management (PIM) surface that can hold any number of products, each with its own variants, pricing, nutrition, quality, images and SEO. The whole point of the module is that you launch each new dairy line with a status flip here rather than a developer deploy.

⚠️
Two save paths — know which button writes where

This screen has two different persistence routes, and it matters which control you use:

• The Add product button and the Recommendations manager write straight to the DOODLY database (real API calls — POST /api/admin/products and PATCH /api/admin/products/[id]). Those changes are permanent, audited and reach every user.

• The per-row Edit button opens the rich 12-tab Product CMS editor. In this build that editor saves your changes to the browser (local overrides via DOODLY_CMS) and overlays them onto the catalogue on the next page load; the very same 12 tabs map one-to-one to the backend's PATCH /api/admin/products/[id] actions (update, pricing, seo, nutrition, quality, status, feature, variant and image operations) that persist to the database in production. When in doubt, use Add product for a brand-new product and the header/Recommendations flows for status and merchandising.

ℹ️
Live vs. demo data — read the banner

Just above the table a status banner appears after load. Green "● Live — N product(s) from the DOODLY database (…)" means you are looking at the real catalogue. If the backend is unreachable you'll see "⚠ Backend offline … — couldn't load live catalogue."; if your role can't view products you'll see "⚠ Your role can't view products (403)."; any other error shows as "⚠ <message>". Confirm the green banner before trusting what the table shows.

The Price column is a variant range, not a single sticker price

Milk is sold per-day per-variant (e.g. a 500 ml daily plan and a 1000 ml daily plan). The Price cell therefore shows the span of those daily prices (lowest–highest), not one number. A product whose variants aren't priced yet — or a "coming soon" product with no active variants — shows "—". To set exact prices, open Edit → Pricing (product-level MRP/selling) and Edit → Variants (per-variant daily/fixed price).

The Status pill is produced by the shared status engine (DOODLY_STATUS), which resolves the product's stored status together with its stock, launch date and variant states into one of these badges. The orderable column is the practical bottom line: can a customer buy it right now?

ℹ️
Two more states you won't set from the pill

DRAFT — a work-in-progress product that isn't shown publicly; used while you build a product before launch. ARCHIVED — the state a product enters when it is soft-deleted (it is hidden and de-listed but never hard-deleted, so history and reports stay intact). Restoring an archived product returns it to DRAFT, not straight to live, so you re-check it before re-launch.

The "flip a status to launch" workflow

This is the headline capability the notice banner promises. Because the storefront reads product status live from the catalogue, changing a product's status is all it takes to launch it (or pull it) — no code change, no deploy.

Workflow — launch a product with a status flip
Create the product (starts COMING_SOON)Fill in pricing, variants, images, copySet status → AVAILABLEStorefront shows it as buyable instantlyCustomers can order it
Figure — from draft to live in one status change; setting AVAILABLE also forces the product Visible.
⚠️
Setting AVAILABLE also makes the product Visible

When a product's status is set to AVAILABLE, the backend automatically forces its Visible flag on, so a freshly launched product is guaranteed to appear on the storefront. The reverse is also true for bulk operations: deactivating hides the product and clears Visible. If a product is "Available" but still not showing to customers, check that it has at least one active, priced, in-stock variant.

How the list loads

When the Products route mounts, the front end calls GET /api/admin/products?pageSize=200&sort=created, maps each returned product into a catalogue row (id = slug, plus name, status, visibility, category, image, featured flag, stock and a selling/MRP-derived price), replaces the in-memory catalogue, re-renders the productAdmin table and shows the green Live banner with the product count and the API base URL. Every mutation on the backend appends a ProductEvent to that product's history, so who changed what (and when) is always recorded.

✅
Best Practice

Treat status as a deliberate release gate. Build a new product fully in DRAFT or COMING_SOON — set its pricing, add and price every variant, upload images, write the description and complete SEO — and only flip it to AVAILABLE once you've clicked the View eye-icon and confirmed the storefront page looks right. Launching is a one-click, instant, customer-facing action, so preview first.

Product price and variant price are different things

The Pricing tab sets the product-level headline price and commercial fields. Best Practice

Curate related products deliberately — pair complementary dairy lines (milk → curd, paneer, ghee) rather than listing everything. Keep the rail short (3–4 items) and lead with your highest-margin or newest line. Use the New flag on a just-launched product and the Best seller flag on your hero SKU so shoppers get honest, useful signals.

What's the fastest way to launch a new product end-to-end?

Click Add product, enter name + slug + a short description, leave status on Coming soon, and save. Then open Edit to set Pricing, add and price Variants, upload Images, and write the Description/SEO. Click the View eye-icon to preview the storefront page. When it's right, set the status to Available to go live.

What's the difference between Status and Visible?

Status is the launch state (Available, Coming Soon, Out of Stock, etc.) and drives whether the product is orderable. Visible is simply whether it is listed on the storefront. Setting Status to Available automatically turns Visible on, but you can independently hide an Available product or show a not-yet-orderable one.

Do products ever get permanently deleted?

No. Deleting a product soft-deletes it — it becomes ARCHIVED, hidden and de-listed, but the record and its history are kept so orders, invoices and reports stay accurate. Restoring returns it to Draft for re-checking before re-launch.

Who can see and change the catalogue?

Only Super Admin and Admin. The Products module isn't granted to any operational role in the default permission matrix, so Support, Operations, Accountant, Delivery Executive, Customer and the rest cannot view or edit products.

How do admin price edits reach shoppers without a deploy?

The storefront reads DB-authoritative commercial fields from GET /api/catalogue and overlays them onto its static presentational data by matching ids. So when you change a variant's price or a product's availability here, the next storefront load reflects it — and the same DB price is what the customer is charged at checkout.
