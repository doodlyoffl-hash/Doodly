Brand Story

Edit the headline copy and packaging-QR of the public “UNFOLD PURE.” story page (/doodly.html) — and download the QR that goes on your milk-bottle labels — without touching any code. Changes go live instantly.

ℹ️
Where this lives

Open it from Admin → Content → Brand Story (URL /admin/brand-story.html). It is a small content-management screen that edits one thing: the storefront brand-story page at /doodly.html (titled “UNFOLD PURE.” — the digital version of the insert printed inside DOODLY packaging). The editor is loaded from, and saved back to, the DOODLY database through the endpoint /api/brand-story. When it loads cleanly you see a green banner: “● Live — brand story loaded from the DOODLY database … Edits persist to Postgres.” A red banner means the backend is offline or your role is not allowed to edit — see Troubleshooting.

ℹ️
What this page is NOT

This is not a full drag-and-drop page builder. The long story panels (Market Reality, Why DOODLY, the farm journey, Trust, contact channels) are fixed in the product and are not edited here. This screen lets you override a small set of headline fields and the QR destination. Everything you leave blank keeps its built-in default.

How editing works

Each field is an override. Type a value to replace the built-in text on the live page; leave a field blank to keep the built-in default, which is shown to you as the field's grey placeholder. There is nothing to “publish” separately — Save changes writes straight to the live page and the DOODLY database. The screen tells you exactly this above the form: “Leave a field blank to use the built-in default (shown as placeholder). Saved instantly to the live page — no code change.”

⚠️
“Products heading” saves locally, but not to the database

The backend deliberately accepts only a fixed whitelist of fields (heroEyebrow, heroTitle, heroLead, heroSub, hookTitle, hookLine, qrDestination, and the three CTA links). Products heading is not on that list. When you save it, it updates the page in your own browser immediately, but the DOODLY database drops it — so after a hard reload (which reloads the copy from the database) or on any other device, the Products heading reverts to its default “Our Premium Collection”. To change it permanently, ask a developer to update the source. Treat this field as a preview, not a permanent setting.

How your changes are stored and applied

Behind the scenes, your edits are not a full copy of the page — they are a small override stored as a single record in the DOODLY database (the “brand story config”, a singleton row). When the public page renders, it starts from the built-in defaults and merges your overrides on top. That is why a blank field simply keeps the default: there is no override to apply. It also means you can never “break” the page by clearing a field — the worst case is that it shows the original text.

ℹ️
Only safe text is stored

When you save, the backend sanitises the payload: it keeps only the recognised fields, and only non-empty trimmed strings. Anything unexpected is silently dropped. This keeps the story page safe no matter what is typed in.

ℹ️
A short cache delay is normal

Because the public story page is cached for speed, a saved change can take a few minutes to appear for everyone. If your own Preview the live page still shows the old text, give it a moment and refresh — the parallel implementation states edits go live “within 10 minutes”.

✅
Best Practice

Keep the QR destination URL pointed at your own /doodly.html story page (the default) unless you are running a specific campaign — that way every bottle scanned lands on the trust-building story, and you can change where it goes later without reprinting. Edit hero copy sparingly and in short, punchy lines (this is a hero, not a paragraph). After any change, click Preview the live page and read it on both desktop and phone before you consider it done. Before a packaging print run, always re-download a fresh QR here so the printed code matches the current destination.

⚠️
A blocked role can still type — but nothing saves

If a role without CMS permission opens this screen, the editor UI still appears, but the backend load fails with a red “Your role can't edit Brand Story (403)” banner and any Save is refused by the server with the toast “Only Marketing / Admin can edit (403).” In that state the person is only changing their own browser copy — the live page and other users are unaffected. Real edits require Super Admin, Admin or Marketing.

Frequently asked questions

Q. Where exactly do my changes appear?

On the public page /doodly.html (“UNFOLD PURE.”). The four hero fields change the very top of that page; the QR destination changes the code in its “Scan & Connect” section and the file you download. Use Preview the live page → to see it.

Q. Do I need to publish or deploy after saving?

No. Saving writes straight to the live page and the DOODLY database — there is no separate publish step and no code change. Allow a few minutes for caching.

Q. What happens if I leave a field blank?

The page uses its built-in default for that field (shown to you as the grey placeholder). Blank never means “empty on the page”.

Q. Can I edit the whole story — the panels about the farm, trust, and products?

Not from this screen. Only the hero copy, the products heading (locally) and the QR destination are editable here. The long narrative panels are part of the product and are changed in code by a developer.

Q. What is the difference between the SVG and PNG downloads?

Both encode the same QR destination. SVG is a vector — it stays razor-sharp at any size, ideal for printing on packaging and large signage. PNG is a high-resolution image — handy for slides, documents and social posts.

Q. If I change the QR destination, do old printed codes still work?

Only if you point the destination at a URL you keep serving. Codes already printed are fixed images pointing at whatever the destination was when they were made. Keeping the destination at your stable /doodly.html lets you change the page's content later without reprinting.
