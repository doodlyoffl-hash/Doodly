Addresses

Where the customer pins the exact GPS doorstep DOODLY delivers to, and manages every saved delivery address — add, edit, set default and delete — with live serviceability enforcement.

ℹ️
Note — two runtimes behind one page

The page /account/addresses.html renders the manifest block addressManager. What the visitor actually sees depends on whether they are truly signed in:

• Signed-in customer (production path): account.js → DOODLY_ACCOUNT.wireAddresses() takes over and drives the real database through /api/addresses. This is Screen A and Screen B below.
• Demo / not-signed-in preview: maps.js → DOODLY_MAPS.mountAddressManager() mounts a local card grid stored only in the browser (localStorage "doodly-addresses"). This is Screen C. The layout code guards this with if (am && !realCust), so demo cards never overwrite a real customer's saved list.

⚠️
Warning — deleting a linked address

If an address is attached to an active subscription, the delete fails and the customer sees a toast such as "Couldn't delete (it may be linked to a subscription)". Reassign or cancel the subscription's address first, then delete.

💡
Tip — the pin does the typing

Dragging the pin (or picking a search suggestion, or tapping "use my location") calls GET /api/geo/reverse and auto-fills pincode, area, city and state, plus house number / street / landmark when the geocoder returns them. The customer can still edit any field — a value they type by hand wins over the auto-fill (the map keeps only the GPS pin).

⚠️
Warning — Save is gated on serviceability

The Save button stays greyed out until the pinned pincode is confirmed serviceable (the internal svOK flag). While the check is in flight it reads "📍 Detecting address…" then "Checking delivery availability…" and Save is disabled. If the location is out of area it turns red — "🔴 Sorry! DOODLY currently does not deliver to this location." — and the box shakes if Save is forced. The backend enforces the same rule again, so a stale UI can never save an out-of-area address.

ℹ️
Note — duplicate guard (new addresses only)

When adding a new address whose house number + pincode already exist in the list, the form asks "You already have an address at this house number and pincode. Save another one anyway?" before continuing. Edits skip this check.

Best Practice

Coach customers (and support agents doing it on their behalf) to drag the pin onto the actual doorstep, not the building centroid, and to fill Landmark, Floor and Door colour. The exact GPS drives live delivery tracking and the executive's route, so a precise pin plus a good landmark is what makes a first-time, on-time delivery.

Serviceability & geo services

The shared checks and endpoints behind the map, the pincode and Save.

Purpose Explain how DOODLY decides whether it delivers to a location, and where the data comes from.
Access Customer (public read) · configured by Admin+ via Serviceable Areas

The pincode is the single primary key for serviceability. Every check (map, typed pincode, and both save endpoints) resolves against the same live ServiceablePincode table (rows that are enabled and not soft-deleted), so an admin adding, disabling or removing a pincode in Admin → Serviceable Areas takes effect immediately with no redeploy.

ℹ️
Note — works with or without a Google key

When a Google Maps key is configured the picker upgrades to the real interactive Google map with Places autocomplete; otherwise it uses DOODLY's built-in dependency-free SVG map and keyless OpenStreetMap (Nominatim) geocoding. The workflow, fields and serviceability gate are identical either way, so the page always works.

Does the customer have to pin a location on the map?

No — the pin is the best path because it captures exact GPS and auto-fills the pincode, but the customer can type the address by hand. If no pin is dropped, the server geocodes the typed address (via OpenStreetMap) so live delivery tracking still has a destination.

Can a customer save an address outside the delivery area?

No. Both the form and the backend require the pincode to match an enabled row in the ServiceablePincode table. An out-of-area address is blocked in the browser and rejected again on the server.

How is the default address chosen?

The first address a customer saves is automatically the default. After that, ticking "Make this my default address" (or the card's "Set default") promotes one address and clears the default flag on all others in a single transaction.

Is there a limit on how many addresses I can save?

The code sets no fixed cap — customers can keep Home, Office, a friend's place and more. Each is validated and serviceable-checked independently.
