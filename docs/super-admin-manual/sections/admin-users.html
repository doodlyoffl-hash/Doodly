User Management

Create staff accounts, assign each person a role, lock or disable access, reset passwords, and soft-delete people who leave — every change is saved to the DOODLY database and written to the audit log.

The User Management page is where you decide who can sign in to DOODLY and what they are. You do not set individual permissions here — those come from the person's role (managed on Admin → System → Roles & Permissions). This page is about the accounts themselves: adding a new team member, changing someone's role, temporarily locking an account, disabling a leaver, forcing a password reset, and removing an account safely. Deletes are always soft — the record is hidden and the account is disabled, but nothing is truly erased, so mistakes can be undone.

The screen is made of two stacked parts that load together: a statistics strip across the top (live counts of your team), and the user directory table below it (one row per account, with all the action buttons). We document each part, then the Create / Edit user form that both the "Create user" button and every row's pencil button open.

ℹ️
Where the data lives

When the DOODLY backend is running, this page reads and writes real accounts through /api/users (Postgres). You will see a green "● Live — N user(s) from the DOODLY database" banner. If the backend is offline you get an amber warning and the page falls back to a local sample list — changes then only affect your browser, not the real system.

⚠️
Super Admin accounts are protected

Only a Super Admin can edit, change the role of, reset the password for, or delete a Super Admin account. If you are an Admin (not Super Admin) and try, the backend returns "Only a Super Admin can manage Super Admin accounts." (403). This stops a lower-privileged admin from tampering with the top account.

ℹ️
Lock vs Disable vs Delete

Lock = quick, reversible freeze (e.g. security concern). Disable = switch off a leaver or dormant account. Delete = remove from the list (still recoverable via Restore, since deletes are soft). Only the person's role — not these status buttons — controls what they can do once signed in.

ℹ️
You never type the new user's password

On Create, DOODLY generates a strong random temporary password automatically and shows it to you once in a pop-up (for example DyKp7mNq2rX7). Copy it and share it with the new person securely — they will be forced to replace it at first login. The password never appears again.

⚠️
Password policy (enforced by the server)

Any password DOODLY sets or a user chooses must be at least 8 characters (max 72) and include an uppercase letter, a lowercase letter, a number, and a special character. The auto-generated temporary password already meets this.

Best Practice

Give every person the lowest role that lets them do their job (least privilege). Reserve Admin and Super Admin for the few who genuinely need them. When someone leaves, Disable or Lock immediately to cut access, then Delete once you have confirmed nothing else depends on the account — because deletes are soft, you can always Restore if you acted too soon. Share temporary passwords over a secure channel, never in a public chat or email thread.

Frequently asked questions

Does deleting a user permanently erase their data?

No. Delete is a soft-delete: the account is set to disabled and stamped with a deletion time, then hidden from the list. Nothing is truly removed, and you can bring it back with Restore. This protects order history and audit trails tied to the account.

What is the difference between locking and disabling an account?

Both stop the person signing in. Lock is meant as a quick, reversible freeze (for example after a security scare) and toggles straight back to active with one click. Disable is for switching an account off more deliberately, such as for a leaver. Choose whichever matches your intent — neither erases anything.

Can I set a user's password myself?

Not from this screen. On Create, DOODLY generates a strong temporary password and shows it once. To change an existing user's password, use Reset pw, which forces them to choose a new password at their next sign-in. You never see or store their real password — that keeps DOODLY secure.

Why can't I change someone's email after creating them?

The email is the account's login identity and must stay unique and stable. When you edit a user, the email field is fixed — you can change the name, role and status, but not the email. If an email is genuinely wrong, create a fresh account and delete the old one.

Who can assign the Super Admin role?

Only an existing Super Admin. An Admin can create and manage most accounts, but any action that grants, changes, resets or removes a Super Admin is blocked with a 403 for anyone below Super Admin.

Where do I actually change what a role can do?

Not here — this page only assigns a role to a person. To change the permissions attached to a role, go to Admin → System → Roles & Permissions (and the Permissions matrix). Every account with that role then inherits the updated access.

⛔
Caution

Never remove or delete the last active Super Admin account — that role is the only one that can manage other Super Admins and unlock the highest-level settings. Always confirm at least one working Super Admin remains before locking, disabling or deleting one.
