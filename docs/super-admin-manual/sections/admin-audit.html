Audit Logs

An append-only, tamper-evident record of every sensitive action across DOODLY — who did what, when, from where and how — with change tracking (old → new value), a live dashboard, powerful search and filters, a per-user activity timeline, sensitive-activity alerts, CSV / Excel / PDF export and configurable retention.

ℹ️
Note — what the Audit Log is (and is not)

The Audit Log is the platform's black box recorder. It captures a rich record for every meaningful event — logins and failed logins, user and role changes, permission grants, order status changes, payments, price and GST edits, report exports, deliveries, procurement and more. Records are append-only: they can never be edited or deleted through the application. There is deliberately no "delete" and no "edit" button anywhere on this screen — the only way records leave the active log is retention/archive, and even archived records can be restored. This is what makes the log trustworthy for investigations and compliance.

💡
How the records get created

You never write audit records by hand. Every module in DOODLY calls one central logger (DOODLY_RBAC.audit() in the browser, lib/auth/audit.ts on the server), which auto-classifies each action into a Module · Action · Entity, extracts any old → new change out of the description, stamps the actor, role, department, IP, browser, OS, device and session, and appends it. In production this screen reads the real Postgres AuditLog table (up to the most recent 2,000 records) through /api/admin/audit-logs and mirrors it into the viewer. A live green banner at the top confirms the source and record count; opening the screen is itself audited as an audit.view event.

⚠️
Who can open this screen

Audit Logs is a System module reserved for the highest trust levels. By default Super Admin has full access and Admin has view-only access; every other role is denied and does not even see the menu item. Exporting the log (CSV / Excel / PDF) and the Retention tab are Super Admin only — an Admin can read and search but cannot export or change the retention policy. The backend enforces the same rule and returns 403 Forbidden to anyone without the auditLogs · view permission.

⛔
Caution — exports are sensitive and are themselves audited

The moment you press CSV, Excel or PDF, DOODLY writes an audit.export record capturing who exported how many records in which format — so the act of taking data out of the platform is permanently on the record. Treat every export as sensitive customer/business data: do not email it, and delete local copies once the investigation is closed. Repeated exports also raise the blue "high export volume" alert on the Dashboard.

Note — the card is strictly read-only

There is no "Edit", "Correct" or "Delete" control on the detail card — only Close. Best Practice — off-boarding checklist

Before you disable a departing employee's account, open their User Timeline and export it. ⚠️
Archiving is not deleting

"Archive old now" and the automatic overflow never destroy data — records move to a separate archive that "Restore archive" can pull back at any time. There is deliberately no way to permanently erase audit records from this screen. Retention only keeps the active log fast; for long-term compliance the manual also notes the design is future-ready for SIEM streaming and compliance reporting off the same append-only writer.

ℹ️
Where retention runs automatically

Retention is also applied every time the Audit Logs screen is opened, so the active log is trimmed to your policy without anyone pressing a button. The two policy numbers therefore quietly govern the whole log's size in the background.

Note — the data behind the screen

In production each row comes from the Postgres AuditLog table (fields: user, role, action, target, IP, device, browser, timestamp), joined to the user's name for display and served read-only from /api/admin/audit-logs (views: list, facets, logins). A companion LoginHistory table records every sign-in success/failure and feeds the failed-login alerts. The API is guarded by the same auditLogs · view permission as the screen and returns 403 to anyone else.

Frequently asked questions

Can anyone edit or delete an audit record to cover their tracks?

No. The log is append-only by design — there is no edit or delete control anywhere in the UI or the API, and even the built-in self-test verifies that no such function is exposed. Records can only be archived (and restored), never erased. That is exactly what makes the log trustworthy as evidence.

What is the difference between "archive" and "delete"?

Archiving moves older records out of the fast active log into a separate store to keep things responsive; "Restore archive" brings them all back. Nothing is destroyed. There is no delete operation for audit data at all.

Why can an Admin see the log but not export it?

Reading the log is low-risk; taking a copy of it out of the platform is sensitive. By default Admin gets the view permission but not export, so only a Super Admin can produce CSV/Excel/PDF files — and every such export is itself recorded as an audit.export event.

Does opening this screen create a record?

Yes. Simply opening Audit Logs writes an audit.view event ("Opened audit logs"), and every export writes an audit.export event. The audit trail even audits the people auditing it.

How far back does the log go?

The active log holds up to your "Max active records" cap (default 5,000) and records within your retention window (default 365 days); the live screen loads up to the most recent 2,000 from the database. Older records sit in the archive and can be restored when you need to investigate further back.

Best Practice — use the log, don't just keep it

Make the Audit Log part of your weekly rhythm, not a box you open only after an incident. A five-minute weekly pass — glance at the Dashboard alerts, spot-check any permission or role changes, and skim exports — turns the log from passive insurance into active early warning. When something does go wrong, you will already know the log inside out.
