Categories

The master list of product groups — Milk, Curd, Paneer, Ghee, Sweets, Bundles — that organises the whole catalogue; every product hangs off a category, the storefront uses them to filter and browse, and this screen is where a Super Admin creates and manages them, live in the DOODLY database.

ℹ️
What a category is, and why it matters

A category is a normalised grouping row in the Category table (id, slug, name, description, image, sort order, active flag). Each product points at one category through its categoryId, so categories are the backbone of catalogue navigation, product filtering and reporting. The screen you see at Admin → Catalogue → Categories is a live card board: when it loads it fetches the real categories from the database and redraws itself, so the tiles you read are the true, current catalogue — never demo data. A green "● Live" banner confirms it.

ℹ️
Categories vs. Products — which screen does what

Use Categories to create and name the groups (e.g. "Dairy products"). Use the neighbouring Products screen to create the actual sellable items and assign each one to a category. The product count you see on a category tile ("4 products") is simply how many products are currently linked to that category. Deleting a category is blocked while any product is still linked to it, so the two screens stay in sync.

Category image is set elsewhere

The database supports an optional image on a category (imageUrl), but this create dialog does not ask for one — new categories start without an image and pick up their icon from the slug. An image can be added later through the category-edit API. The slug is set once at creation and is not editable afterwards, so choose it carefully.

Category Lifecycle — Edit, Hide & Delete

How a category changes over its life: renaming, re-ordering, hiding from the storefront, and the guarded rules around deleting one.

Purpose Document the full set of operations the categories API supports so staff understand what can be changed after creation and what the safety rails are.

ℹ️
Editing & deleting run through the API

Today the on-screen board offers view and Add category. Renaming, re-ordering, hiding and deleting a category are backed by the /api/categories/{id} endpoint and are performed by staff with the right permission (typically via the product/catalogue tooling). The most common day-to-day change — hiding a category from customers — is simply setting its status: create or update it with status Hidden, and the storefront stops listing it while staff still see it on the board marked "hidden".

⚠️
The slug can't be changed after creation

Renaming, re-ordering, hiding and image changes are all allowed, but the slug is fixed for life — the update endpoint deliberately does not accept a new slug, because links and product associations depend on it. If you truly need a different slug, create a fresh category with the correct slug, move the products across on the Products screen, then delete the old (now-empty) one.

⛔
Delete is blocked while products are linked

You cannot delete a category that still has products in it — the API rejects it with "Reassign or remove its products before deleting this category." First open the Products screen and move every product to another category (or remove them), so the tile reads "0 products". Only then will the delete succeed. Deletion is permanent; there is no undo, so most teams simply Hide a category they no longer use rather than delete it.

ℹ️
Categories is an Admin-and-above module

In the permission matrix only Super Admin and Admin hold the categories module (Super Admin has everything; Admin holds it at "full", i.e. view / create / edit / delete). Marketing, operations, support, procurement, quality, inventory, accountant and delivery roles have no categories access and never reach this screen. The one exception is the read side: the public list endpoint returns active categories to anyone, because the storefront needs them to build its catalogue navigation. Server-side permission checks are the real security boundary — the menu filter is convenience only.

✅
Best Practice — plan the slug and the order before you save

Because the slug is permanent, decide it deliberately: keep it short, lowercase and hyphenated (e.g. dairy-products, not Dairy_Products), and matching how you'd want it to read in a link. Set a sensible Display order in tens (10, 20, 30…) so you can slot new categories between existing ones later without renumbering everything. Create a category as Hidden first, add and check its products on the Products screen, then flip it to Active — that way customers only ever see a category once it actually has products in it.

Frequently asked questions

Q. What is the difference between "Hidden" and "Delete"?

Hidden keeps the category and all its product links intact but stops the storefront from listing it — staff still see it on the board marked "hidden", and you can flip it back to Active any time. Delete removes the category permanently and is only allowed once no products are linked. For anything you might reuse, hide it; only delete truly obsolete, empty groups.

Q. Can I change a category's slug later?

No. The slug is set when you create the category and cannot be edited afterwards, because product associations and links rely on it. If you need a different slug, create a new category, re-assign the products to it on the Products screen, and delete the old empty one.

Q. Where does the product count on a tile come from?

It's a live count of how many products are currently linked to that category (their categoryId points at it). Assigning or moving products on the Products screen changes the number the next time this board loads.

Q. Why can customers see categories if it's an admin module?

The management of categories is Admin-only, but the read of active categories is public so the storefront can build its catalogue menu and filters. Hidden categories are never sent to customers — only staff with categories:view receive them.

Q. Do I set the category image here?

Not in the Add category dialog — new categories start without an image and use an icon derived from their slug. A category image (imageUrl) can be attached later through the category-edit endpoint.

Q. Is every change to a category recorded?

Yes. Creating, updating and deleting a category each write an audit entry (category.create / category.update / category.delete) to the platform Audit Logs, so there is always a trail of who changed the catalogue structure and when.
