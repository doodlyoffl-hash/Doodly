Delivery Settings

One control panel that decides which dates a customer can pick to start milk delivery — the nightly order cut-off, the morning delivery window, how far ahead an order may be booked, which weekdays you run, and any holidays or maintenance days you close.

ℹ️
What this screen controls

Every rule you set here becomes the single source of truth for the start-date picker used everywhere on the storefront — the checkout, the "choose your first delivery day" calendar, and the subscription "change start date" panel. A brand-new employee can think of it as: "the settings that tell customers which mornings we can deliver." There is no code change and no redeploy — you edit the form, click Save settings, and the storefront honours the new rules immediately, platform-wide.

ℹ️
Where the data lives

All values are stored in one database row — the DeliveryConfig singleton (its id is literally "singleton", so there is exactly one shared config for the whole business). The row is created automatically with sensible defaults the first time the page is opened. The screen reads it with GET /api/admin/delivery/config and writes it with PATCH /api/admin/delivery/config. Every save is diffed old → new and written to the audit log.

Live vs local

On open, the page fetches the config from the database and shows a green banner: "● Live — delivery configuration from the DOODLY database (<api-base>). Saves apply platform-wide." If the backend cannot be reached it shows a red banner "⚠ Backend offline at <api-base> — settings save locally only." and any edits stay in this browser only until you re-save with the backend up. This manual documents the live (database-backed) behaviour.

⚠️
Minimum must never exceed maximum

If you set Minimum advance greater than Maximum advance, the server rejects the save with a 400 error: "Minimum advance days cannot exceed maximum advance days." Keep minimum ≤ maximum (for example min 1, max 30).

ℹ️
Holidays vs Blackout dates

Both lists block delivery on the dates you list. They exist as two boxes only to keep your records tidy — Holidays for planned public holidays and Blackout dates for maintenance / operational closures. On the storefront they simply show as unavailable days (reason "holiday" vs "blackout"). A date in either list is unbookable.

⚠️
If your role cannot edit

If someone without edit rights clicks Save, the change is kept in their own browser but the database rejects the write (403). They see the toast "Saved locally — your role can't edit settings (403)" and nothing changes for other users. Only Super Admin, Admin, and Operations can make settings stick platform-wide.

ℹ️
This screen edits scheduling only

The DeliveryConfig row also stores a few operational values — delivery charge, free-delivery threshold, auto-assignment on/off, per-executive and per-trip bottle capacity, and the late-delivery SLA time and grace. Those are not exposed on this form; they are managed from the Operations → Delivery Mgmt, Auto Assignment and Late Deliveries screens. The Delivery Settings page saves only the ten scheduling fields listed above and leaves the operational fields untouched.

✅
Best Practice

Make scheduling changes ahead of time, especially the cut-off and holiday lists, so customers who are mid-checkout aren't surprised. After every save, confirm you got the green "…live platform-wide" toast, then open the storefront date picker in another tab and check the affected date shows the state you expect (available, or blocked with the right reason). Add holidays in bulk at the start of each month rather than one at a time.

Frequently asked questions

Q: Does changing the cut-off time reschedule customers who already subscribed?

A: No. The cut-off and advance-day rules only decide the earliest start date the picker offers on new orders and start-date changes. Active subscriptions already have their dates and are not moved.

Q: What happens if I un-tick every delivery day?

A: The form never lets the list be empty — if no day is ticked it falls back to all seven days. An empty set would make every date unbookable, so it is prevented on purpose.

Q: The weekday boxes have Saturday and Sunday ticked, but weekend delivery is off. Which wins?

A: The weekend switch wins. "Weekend delivery" off removes Saturday and Sunday from the effective days even when their individual boxes are checked — treat it as a quick override.

Q: Is every change tracked?

A: Yes. Each Save writes an audit entry (delivery.config.update) that names the exact fields that changed and records who changed them (your role), so the history is fully accountable.
