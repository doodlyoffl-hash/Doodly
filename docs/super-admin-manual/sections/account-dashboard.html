Customer Dashboard

The signed-in customer's home screen — a single, live "everything about your milk today" view that summarises the subscription, next delivery, wallet, reward points, referrals, orders, invoices, bottles, the loyalty tier card, the Monthly Puzzle Challenge and a preview of recent orders, every tile deep-linking to its full page.

ℹ️
Who this screen belongs to

This is a customer-facing page inside the "My Account" surface (URL /account/dashboard.html), not an admin console. It appears in this manual so Super Admins, Managers and Support staff understand exactly what a customer sees, how it is fed with data, and how to explain it or reproduce a complaint. Every number is computed live from the signed-in customer's own records — a staff member cannot edit anything here; they observe it (for example while helping over the phone or via impersonation).

Block recipe (for reference)

The page is assembled from these blocks, in this order: page head → customerKpis (page "dashboard") → quickActions → loyaltyCard → puzzleCard → customerSub → table (dataset "orders", toolbar off, pager off). Each block renders an empty mount that is then filled with live data by its module (Customer, Loyalty, Puzzle) after the account data is fetched.

Region A — Greeting (page head)

The recipe seeds the heading as "Good morning 👋" with the subtitle "Here's everything about your milk today." As soon as the signed-in customer's data is loaded, the greeting is rewritten to the correct time of day and personalised with the customer's first name.

💡
Tip — how the numbers are derived

The core figures (name, wallet balance, loyalty points, bottles pending, next delivery, active subscription and the order/delivery/invoice counts) come from one backend call, GET /api/account/summary. Referral earnings, active-order counts, skipped-delivery counts and the "latest invoice" line are computed on the device from the customer's own hydrated order, delivery, invoice and referral records. Tier is derived from points (Silver below 500, Gold 500–1499, Platinum 1500+).

Note — the "Per delivery" and "Renews" rows are conservative

Those two rows are only drawn when the live subscription supplies a real per-delivery price and end date. If a figure isn't available, the row is omitted rather than guessed — so a customer is never shown an invented price or renewal date. The exact figures always live on the Subscription page.

SituationWhat the customer sees
Signed-in customer, backend healthyAll cards populate with live figures; greeting shows real name + time of day.
Section fetch fails (e.g. referrals) but profile loadsThat one section stays empty/zero; the rest of the dashboard is normal — no error is forced.
Session expired / revoked (401)The dashboard drops the stale session and redirects to /login/customer.html?expired=1 so the customer signs in again. Demo data is never shown.
Backend unreachable / timeout while signed inHonest empty states everywhere plus a connectivity banner; the page never falls back to the sample "Ananya" account.
Not signed in (public/demo)Access is guarded — the customer is prompted to log in or create an account before the account pages open.
✅
Best Practice

When a customer says "my dashboard is empty," first confirm whether it is a new account with no orders yet (genuine empty states, no banner) or a connectivity banner (backend unreachable). The two look different: empty states carry friendly "get started" buttons, while the connectivity banner is a fixed pill telling the customer we're offline. Only the second one is an incident.

⚠️
Warning — this screen is read-only for staff

Nothing on the dashboard can be edited from here. Changes to a subscription, wallet, rewards or bottles happen on their own pages (and, for staff, in the Admin modules). If a KPI looks wrong, correct the underlying record in the right module — the dashboard will recompute the next time it loads.

Is the customer dashboard the same as the admin dashboard?

No. This is the customer's own "My Account" home. The admin operations dashboard is a separate screen in the Admin surface with business-wide KPIs and charts.

Can a staff member change anything from this screen?

No. The dashboard is a read-only summary. All edits happen on the dedicated pages or in the Admin modules; the dashboard recomputes from those records on the next load.

Why does "Upcoming Delivery" always say "before 7 AM"?

DOODLY runs a single early-morning delivery run, so the promise is a cut-off ("before 7 AM"), not a time slot. The card shows the next scheduled delivery's date with that promise, or "No delivery scheduled" when there is none.

Why can the same customer see different tiles than another customer?

Cards are data-driven. The Pure Rewards and Puzzle cards only appear when their programmes are enabled and relevant; the subscription card switches to an empty state without an active plan; and captions like "N unpaid invoices" reflect that customer's real records.
