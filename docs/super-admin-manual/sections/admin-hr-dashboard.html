HR Dashboard

A single, read-only snapshot of the whole workforce — headcount, who is present or absent today, who is on leave, how many payslips still need to be run, how much salary-advance money is still owed, and how staff are split across departments — pulled live from the same records that power Employees, Attendance, Leave, Salary Advances and Payroll.

ℹ️
What this screen is (and is not)

The HR Dashboard item under Admin → Human Resources is served by the storefront admin page /admin/hr-dashboard.html, which mounts the DOODLY_HR module's mountDashboard() renderer into #hrDashboardMount. It is a reporting screen only — there are no forms, no filters, no buttons and nothing to save. Every number is calculated fresh by the backend each time the page loads, from one call to GET /api/admin/employees?view=dashboard (server function hrDashboard()). To change any of the underlying figures you go to the relevant HR section — the dashboard just summarises them. Money (the "Advances outstanding" figure) is shown in ₹; the backend stores it as integer paise (₹1 = 100 paise).

ℹ️
Who can open it

The API gate is requirePermission(req, "employees", "view") — access is granted by the Employees module permission, not a separate one. In the default role matrix that means Super Admin and Admin (full access to everything HR) and Accountant (view-level access to Employees, HR Dashboard, Attendance and Leave, plus deeper rights on Payroll and Advances). No other staff role — Operations, Procurement, Inventory, Quality, Marketing, Support, Delivery Executive — can open HR at all, and Customers never can. Server-side permission checks are the real boundary; hiding the menu item is only convenience.

Every KPI card and where the number comes from

Cards render in a fixed order left-to-right, top-to-bottom. Each card shows a large number (or a ₹ amount) with a small label under it. A colour tone is applied to draw the eye: green = healthy, amber = needs attention, red = loss of headcount. All counts ignore soft-deleted (removed) employees — every query filters deletedAt: null.

⚠️
"Absent today" is an estimate, not a headcount of absences

Read this card carefully. It is computed arithmetically as Active − Present today − On leave (floored at zero). It does not read attendance rows marked ABSENT. So early in the day, before attendance has been marked, this number will be large simply because few people are marked present yet — it will shrink as staff are marked in. It also treats employees whose status flag is ON_LEAVE as "on leave", but not those merely marked PAID_LEAVE/SICK_LEAVE in today's register. For the true, per-person absence picture use Admin → Human Resources → Attendance, which counts real ABSENT marks for the chosen date.

Department-wise headcount

Below the KPI cards sits one panel titled Department-wise headcount. It lists how many people belong to each department, so you can see where the team is concentrated. When there are no employees yet it shows the plain message "No employees yet." A short footer line under the panel reminds you that the detailed work lives elsewhere: "Attendance, leave, salary, advances and payroll are managed under Human Resources → their sections."

ℹ️
The known departments

Each employee is filed under one of ten standard departments defined in the module: Delivery, Procurement, Production, Quality, Inventory, Customer Support, Finance, Marketing, Operations, Administration. The headcount table only lists departments that actually have staff, so a brand-new business may show just two or three rows.

ℹ️
The API also returns an employment-type breakdown

Behind the scenes the same hrDashboard() response includes a byType array (headcount split by Full Time / Part Time / Contract / Internship). This screen does not draw it — it renders only the eight KPI cards and the department table — but the figure is available to other tools and reports.

Loading and error states

While the figures are being fetched the page shows a skeleton placeholder (a shimmering panel). If the request fails — no network, session expired, or no permission — the whole area is replaced by an amber warning box reading "Couldn't load the HR dashboard." (or the specific server message). There is no auto-refresh: the numbers are captured at page load, so reload the page to get the latest figures.

✅
Best Practice — read it in the right order, at the right time

Check the HR Dashboard after the morning attendance is marked, not before — otherwise "Present today" is artificially low and "Absent today" artificially high. Treat the two amber cards (Absent today, On leave) and the red card (Resigned / exited) as your daily watch-list, and the two work-queue cards (Payroll pending, Advances outstanding) as your month-end to-do list: a non-zero "Payroll pending" means draft payslips still need finalising, and a large "Advances outstanding" is money you are still recovering. When a number looks wrong, don't fix it here — open the matching HR section, correct the source record, then reload the dashboard.

Frequently asked questions

Can I edit anything from the HR Dashboard?

No. It is entirely read-only — there are no forms, buttons, filters or exports. It only reports. To change a figure, open the underlying section (Employees, Attendance, Leave, Salary Advances or Payroll), make the change there, then reload the dashboard.

Does "Total employees" include people who have left?

Yes. "Total" counts everyone still on the books (not soft-deleted), including On-leave, Resigned and Terminated staff. Use the "Active" card for the number of currently-working people, and "Resigned / exited" for those who have left.

Why doesn't "Present + Absent + On leave" always add up to "Active"?

Because "Present today" counts attendance rows (Present, Half-day or WFH) which can include employees whose profile status isn't strictly Active, while "Absent today" is floored at zero and derived arithmetically. The three cards are indicators, not a strict reconciliation. For an exact daily tally, use the Attendance register.

Is the ₹ figure in rupees or paise?

The card shows rupees. Internally the API returns advancesOutstandingPaise in paise and the screen divides by 100 for display, so ₹18,500 is stored as 1,850,000 paise.
