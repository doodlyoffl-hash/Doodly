Inventory

The single stock ledger for everything DOODLY holds on hand — filled milk bottles ready to sell and the raw materials and supplies that keep the dairy running — showing live on-hand counts, what is reserved for open orders, a colour-coded health status, and a one-click way to correct any count with an audited reason.

ℹ️
One screen, two stock sources

The Inventory view combines two kinds of stock into one table so staff never have to look in two places. (1) Filled-bottle products come from each product Variant — its stock (on hand) and reservedStock (held for open orders). (2) Raw materials and supplies — empty bottles, foil caps, sanitiser and so on — come from InventoryItem rows and their quantity. Each row is tagged internally as a variant or a material so the same "Adjust stock" tool works on both. The overview reads GET /api/admin/inventory/overview; a green banner ("● Live — N inventory item(s) from the DOODLY database") confirms the numbers are live, not the demo seed.

ℹ️
How the status colour is decided

Every row gets a single traffic-light Status computed from its numbers, so you can scan the whole page and instantly see what needs attention. The rule (in statusOf) is applied in order: Available (stock − reserved) is 0 or less → Out of stock (red); else stock is at or below its Reorder at level → Reorder (red); else stock is at or below twice the reorder level → Low (amber); otherwise → Healthy (green). For raw materials, reserved is always 0, so their status is judged on quantity versus reorder level alone.

More is tracked than the five visible columns

Behind each row the overview also carries Available (stock − reserved), Reserved (held for open orders, always 0 for supplies), the item Unit (e.g. "1000 ml", "pcs"), a stock value in paise (variant stock × its daily/fixed price; supplies are valued at 0), and the last updated time. The whole-inventory stats block also totals items, units, available, reserved, low-stock, out-of-stock and inventory value — the KPI row surfaces the four that matter most day to day.

⚠️
Stock can never go negative

If a Decrease, Issue or Damage would take the count below zero, the server rejects it with the exact current figure, e.g. "Adjustment would make stock negative (current 80)." — the count is left untouched. To zero an item deliberately, use Set exact count with a quantity of 0. Every applied adjustment is permanent in the audit trail; there is no "undo" other than a compensating adjustment.

✅
Prefer "Adjust stock" over editing the quantity directly

You can change an item's quantity with a PATCH, but the Adjust stock tool is the better path: it forces a reason, records the exact before → after, refuses negatives, and writes a typed inventory.adjust.<mode> event — so the why is never lost. Use the edit path for the item's descriptive fields (name, unit) and its reorder level; use Adjust stock for the count.

ℹ️
Inventory is a supply-side module

In the permission matrix the Inventory Manager role holds this module at "full" (so it can view, adjust, create, edit and delete), Super Admin and Admin have full access to everything, and Operations gets it at "view" only — it can watch stock levels but not change them. Support, procurement, quality, marketing, accountant and delivery roles have no inventory access and never see this screen. Every API call runs requirePermission(inventory, …) on the server, so hiding the menu item is convenience only — the server is the real boundary. Even if a view-only user reaches the Adjust stock button, applying the change is rejected with a 403.

✅
Best Practice — count, then reconcile with a reason every time

Run a physical cycle count on a rota, then reconcile the system to it through Adjust stock — use Set exact count when you know the true figure, or Damaged / write-off for known breakage — and always write a specific reason ("morning cycle count", "3 crates broken in transit"). Set each item's Reorder at to about half of your safe minimum so the amber Low band gives you lead time before it turns red. Watch the Reorder now KPI daily; when it is above zero, use the Reorder filter chip to see exactly which items to purchase.

Frequently asked questions

Q. What's the difference between "Stock" and "1000 ml ready"?

Stock in the table is everything physically on hand. The 1000 ml ready and 500 ml ready KPIs show the available figure — stock minus whatever is already reserved for open orders — so they tell you how many bottles you can still sell right now.

Q. Why did an item turn red even though it still has stock?

Because its Stock has fallen to or below its Reorder at level (that's the Reorder state), or its available count hit zero because everything is reserved (that's Out of stock). Red means "act now", not necessarily "empty".

Q. I received a delivery of caps. How do I add them?

Click Adjust stock, pick the caps item, choose Increase / receive, enter how many arrived, type a reason like "delivery received", and press Apply adjustment. The count goes up and the change is logged.

Q. Can I undo an adjustment I made by mistake?

There is no delete or undo — the ledger is append-only for integrity. Post a compensating adjustment instead (e.g. if you added 500 too many, do a Decrease / issue of 500, or use Set exact count to fix it to the true number) and note the correction in the reason.

Q. Does adjusting a bottle's stock affect what customers can order?

Yes — filled-bottle rows are the product Variant stock, so lowering a variant's count reduces what's sellable, and its status will flip to Low / Reorder / Out of stock accordingly. A bottle adjustment also writes a STOCK entry to that product's event history, unlike a supply-only item.
