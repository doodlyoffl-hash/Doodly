Payroll

Run the monthly salary for the whole team in one click: DOODLY reads each person's attendance and salary structure, works out earnings, deductions and net pay, produces a printable salary slip per employee, recovers any salary advance from the pay, and prepares a bank-transfer report — moving each payslip through Draft → Finalized → Paid.

ℹ️
What this screen is (and where it lives)

The Payroll item under Admin → Human Resources is served by the storefront admin page /admin/payroll.html, whose manifest block { type:"hrPayroll" } mounts a single empty #hrPayrollMount container. The DOODLY_HR module's mountPayroll() renderer (in assets/js/employees.js) fills it and talks to one backend endpoint, /api/admin/payroll (server code in next-app/lib/payroll/service.ts). Every rupee shown on this page is stored by the backend as an integer number of paise (₹1 = 100 paise) and only divided by 100 for display, so the money is always exact — there is no floating-point rounding of your salaries.

ℹ️
Who can open it, and who can run payroll

The endpoint is gated per action: reading the list, a slip, or the bank report needs requirePermission(req, "payroll", "view"); generating, finalizing or marking a payslip paid needs requirePermission(req, "payroll", "edit"). In the default role matrix that means Super Admin (all access) and Admin (full) can both view and run payroll, and the Accountant role also has payroll: full — so an Accountant can generate, finalize and pay too. No other staff role (Operations, Procurement, Inventory, Quality, Marketing, Support, Delivery Executive) has any Payroll rights, and Customers never do. If a user only has view rights, the front-end hides the Generate payroll button and the per-row Finalize / Mark paid buttons — but the server-side permission check is the real boundary.

⚠️
Payroll depends on two other HR screens

A payslip can only be produced for an employee who has (1) an active salary structure — set on the Employee detail / Salary structure, defining Basic, HRA, Conveyance, Special allowance, other earnings, Professional Tax and other deductions — and (2) their attendance marked for that month in HR → Attendance. Employees with no active salary structure are silently skipped when you generate. Advance recovery pulls from approved requests in HR → Salary Advances. So the correct month-end order is: mark attendance → confirm salary structures → generate payroll → review → finalize → pay.

Generate Payroll

The one-click monthly run that builds a draft payslip per employee from attendance + salary structure.

Purpose Turn a month of attendance and each person's salary structure into draft payslips, automatically — proration for absences, overtime, planned advance recovery and fixed deductions all worked out for you.
Access Super Admin Admin Accountant — edit Path Admin → Human Resources → Payroll → Generate payroll

Clicking Generate payroll from the register (after a confirm dialog) sends action:"generate" with the selected month. The backend loops every eligible employee, computes a payslip, and saves it as a Draft. It is idempotent: running it again recalculates existing drafts in place but never touches a payslip that is already Finalized or Paid, and never creates duplicates.

ℹ️
Who gets a payslip when you generate

Generate-all builds slips for every employee whose profile is not removed (deletedAt = null) and whose status is ACTIVE or ON_LEAVE. Anyone without an active salary structure is skipped (counted in the "skipped" toast). Anyone whose payslip for that month is already Finalized or Paid is left untouched. The toast reads, for example, "Generated 34 payslip(s), 2 skipped (no salary structure)".

The request fields

From the admin button, only month is sent (all active employees, no ad-hoc bonus). The underlying API also accepts three optional fields, used when payroll is run for a single employee (for example from tooling or a future per-person action).

How each payslip is calculated

For every employee the engine pulls the month's attendance and the active salary structure, then works top-to-bottom. This is the exact logic that fills the salary slip; understanding it makes the numbers self-explanatory.

LineHow it is worked out
Working daysThe number of calendar days in the month (e.g. 31 for July).
Loss-of-pay (unpaid) daysAbsent days + (Half-days × 0.5). Paid leave, sick leave, weekly-off and holiday do not reduce pay.
Proration factorPaid days ÷ Working days, where Paid days = Working days − LOP days. Every fixed earning below is multiplied by this factor.
Basic, HRA, Conveyance, Special, Other earningsEach taken from the salary structure and prorated by the factor above.
Overtime pay(Overtime minutes ÷ 60) × hourly Basic × 1.5, where hourly Basic = monthly Basic ÷ (Working days × 8). Overtime minutes come from the attendance records.
Bonus / IncentiveAdded if supplied (single-employee run only); otherwise ₹0.
GrossSum of all the earnings above.
Advance recoveryFor each of the employee's approved advances still owing, the smaller of its installment amount and its remaining balance is planned for recovery this cycle. (Actually deducted from the advance ledger only when the slip is marked Paid.)
Professional Tax + Other deductionsFixed amounts from the salary structure.
Total deductionsAdvance recovery + Professional Tax + Other deductions.
Net paymax(0, Gross − Total deductions) — never negative.
💡
Tip — re-generate safely, as many times as you like

If you fix an attendance mark or update a salary structure after generating, just click Generate payroll again for the same month. Every Draft is recomputed from the corrected data, so the drafts always reflect the latest records. Only once you Finalize a slip is it frozen against re-generation — that is the point of finalizing.

What "Mark paid" actually does

Marking a Finalized payslip as Paid runs inside a single database transaction so the money and the advance ledger always stay in step:

StepEffect
Recover advancesIf this slip has planned advance recovery and hasn't been recovered before, the engine walks the employee's approved advances oldest-first and, for each, deducts the smaller of its installment, its remaining balance, and what's still to recover. Each deduction creates an AdvanceRecovery record, reduces the advance's remaining balance, and marks the advance Completed when it hits zero. Recovery is applied once per slip (idempotent).
Stamp paidThe payslip status becomes PAID and paidAt is set to now.
Notify the employeeSends a "Salary credited 💰" notification with the month, the net amount and the payslip code (non-blocking — payment still succeeds if the message fails).
AuditWrites a payroll.paid audit entry (employee code, slip code and net amount). Finalize writes payroll.finalize; generate writes payroll.generate.
⚠️
Finalize before you pay — and check the order of operations

The register only offers Mark paid on a Finalized slip, so you must Finalize first. Finalizing is what freezes the figures against a later re-generate, so if you spot a mistake on a Draft, fix the source (attendance / salary structure / advance) and re-generate before finalizing. Marking Paid a second time is harmless (it returns without re-recovering advances), but there is no "un-pay" button — treat Mark paid as the point of no return.

⛔
Caution — paying recovers advances for real

Advance recovery is only planned at generate time; it is actually applied to the salary-advance ledger the moment you Mark paid. That reduces the employee's outstanding advance balance and can complete the advance. There is no reversal in the UI — if you paid in error, you must correct the advance and pay records manually. Confirm the net amount and the "Advance recovery" line on the slip before clicking Mark paid.

✅
Best Practice — the month-end payroll checklist

Run payroll in this order every month: (1) close and correct Attendance for the month so proration and overtime are right; (2) confirm every active employee has an up-to-date salary structure and that pending advances are approved; (3) click Generate payroll and read the "N generated, M skipped" toast — investigate any "skipped (no salary structure)"; (4) open a few Slips and sanity-check gross, deductions and net; (5) re-generate if you had to fix anything; (6) Finalize the slips; (7) open the Bank report, transfer the money through your bank, then (8) Mark paid so advances are recovered and staff are notified. Keep the CSV export as your internal record of each run.

Frequently asked questions

If I run "Generate payroll" twice, do employees get paid twice?

No. Generate is idempotent — it recalculates each existing Draft in place and never creates duplicates, and it never touches a Finalized or Paid slip. Money only moves when you Mark paid, and marking an already-Paid slip does nothing.

How are absences and overtime reflected in pay?

Fixed earnings (Basic, HRA, etc.) are prorated by "paid days ÷ working days", where unpaid days are absences plus half of any half-days (paid/sick leave, weekly-off and holidays are not unpaid). Overtime is paid separately at 1.5× the hourly Basic rate for the recorded overtime minutes.

When exactly is a salary advance recovered?

The recovery amount is planned (shown as "Advance recovery" on the slip) when payroll is generated, but it is only applied to the advance ledger when you Mark the slip Paid — reducing the outstanding balance and completing the advance if it reaches zero. It is applied once per slip.

Can an Accountant run payroll, or only Admins?

The Accountant role has payroll: full in the default matrix, so an Accountant can generate, finalize and mark paid — the same as Admin and Super Admin. No other staff role can open Payroll at all.

What is the payslip code format?

Each slip is coded PS-<month>-<sequence>, e.g. PS-2026-07-0004 — the month plus a four-digit running number within that month.

Are the ₹ amounts stored as rupees or paise?

The screen shows rupees, but the database stores every amount as integer paise (₹1 = 100 paise), which is why the totals are always exact. The CSV export writes rupees (paise ÷ 100).
